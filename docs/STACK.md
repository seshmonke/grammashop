# Стек проекта

Референс-документ. Здесь фиксируем принятые решения по технологиям, чтобы не
держать их в голове и не переспрашивать через месяц, почему выбрали именно
это.

## Frontend
**React + TypeScript + Vite**
Стандартный, самый живой экосистемный выбор. Vite — минимум конфигурации.

## UI-кит
**shadcn/ui** (Radix UI + Tailwind), компоненты копируются в проект, а не
ставятся как отдельная рантайм-зависимость.
Radix даёт доступность и поведение "из коробки" (модалки, дропдауны,
тултипы), Tailwind — полный контроль над стилями. Подходит и для витрины
(глубокая кастомизация под брендинг продавца, включая кастомный фон
витрины на Тарифе 3), и для обеих админок — один и тот же кит
переиспользуется между ними (см. «Роутинг»). Соответствует принципу
"минимум зависимостей": нет vendor lock-in на чужой API компонентов.
Минус: сложные виджеты (таблицы с сортировкой/пагинацией и т.п.) не идут
готовыми "из коробки" — либо пишем сами, либо добавляем точечно headless-
библиотеки вроде TanStack Table поверх.

### Вёрстка

Адаптивная, кроссплатформенная, mobile-first. Telegram Mini Apps открываются
преимущественно с телефона (витрина — почти всегда, админки — часто с
телефона у продавца), поэтому вёрстка проектируется сначала под мобильный
экран, затем расширяется под планшет/десктоп через Tailwind-брейкпоинты
(`sm:`/`md:`/`lg:`), а не наоборот. Кроссплатформенность — учитываются
различия между Telegram-клиентами (iOS/Android/Desktop/Web): safe-area
отступы, поведение `viewport`/скролла внутри Mini App.

### Дизайн-направление

**Y2K** (хром, магента, ice, жирная геометричная типографика Unbounded)
— зафиксировано в диалоге 19.07.2026, Спринт 10. Полная спецификация
токенов, типографики, приёмов и символа/логотипа — отдельный документ:
[docs/design/DESIGN_SYSTEM.md](design/DESIGN_SYSTEM.md), лого-ассет —
[docs/design/logo.svg](design/logo.svg). Здесь — только принципы,
которые не меняются от смены визуального направления:

- Тема — из Telegram `themeParams`: light/dark и акцентный цвет клиента
  подхватываются, ТМА ощущается нативной частью Telegram. Брендовые
  токены (палитра/тип из design-system) — акцент **поверх** этой темы,
  не замена её.
- Акцент — дефолт витрины продавца (Тариф 1) и стартовое состояние
  Тарифа 2/3 до кастомизации продавцом (см.
  [CONCEPT.md#тарифы](CONCEPT.md#тарифы)); неизменен только там, где
  витрина продавцу не принадлежит — обе админки, аватарка бота,
  будущий лендинг.
- Blur/glass-эффекты (если используются) — не на карточках товаров:
  на каждом элементе скролл-списка лагает в Telegram WebView на
  бюджетных Android — основной аудитории. Ограничение платформы, а не
  вкусовщина.
- Админки (продавцовская и платформенная) — без декора: плоско, плотно,

#### Сетка каталога — кросс-платформенная вёрстка

Решено 20.07.2026, при планировании Спринта 17 (перевод витрины на
Y2K + сетка каталога, см. [TASKS.md](TASKS.md)). ТМА открывается не
только в мобильном Telegram — десктопный клиент Telegram тоже рендерит
Mini App, и его окно пользователь резайзит произвольно (это не
фиксированный набор из 2-3 стандартных устройств, как у обычного сайта
с постоянной аудиторией). Раскладка каталога должна сама подстраиваться
под ширину, а не только под мобильный вьюпорт.

**Флюидная CSS Grid вместо фиксированных брейкпоинтов**:

```css
grid-template-columns: repeat(auto-fill, minmax(var(--card-min), 1fr));
```

Число колонок считается самим браузером из доступной ширины — 2 на
узком мобильном, органично растёт до 4-6+ на широком экране, без
явных `@media`-порогов на каждый шаг. Источники (не по памяти,
подборка 20.07.2026):
- [CSS Grid Best Practices + шаблоны, TestMu AI](https://www.testmuai.com/blog/css-grid-best-practices/) —
  `repeat(auto-fill, minmax(280px, 1fr))` как база; `auto-fill` держит
  пустые треки, если элементов не хватает на полный ряд, `auto-fit`
  стягивает оставшееся место на существующие — для каталога, где
  количество карточек меняется, `auto-fill` предсказуемее (не
  растягивает единственный ряд карточек на всю ширину).
- [MDN — Realizing common layouts using grids](https://developer.mozilla.org/en-US/docs/Web/CSS/CSS_grid_layout/Realizing_common_layouts_using_grids) —
  тот же паттерн как канонический способ карточных списков без
  медиа-запросов.
- [NN/g — Using Grids in Interface Designs](https://www.nngroup.com/articles/using-grids-in-interface-designs/),
  [UXPin — UI Grids guide](https://www.uxpin.com/studio/blog/ui-grids-how-to-guide/) —
  общий принцип адаптации числа колонок под ширину экрана и
  ограничение верхней плотности (не больше ~8-10 колонок — визуальный
  шум); нам не подходит буквально (это про 12-колоночные системы для
  макетов страниц), но подтверждает верхнюю границу: широкий монитор
  не должен разъезжаться в десяток мелких плиток.

Два параметра держат сетку в разумных границах на обоих концах шкалы,
подбираются эмпирически при реализации (тем же приёмом, что уже не раз
спасал на этом проекте — Playwright-сверка на сетке вьюпортов
320-1920px, не на глаз, см. историю Спринта 15):
1. **`--card-min`** — нижняя граница трека, гарантирует минимум 2
   колонки на самом узком поддерживаемом экране (см. запись Спринта
   17 в TASKS.md).
2. **`max-width` внешнего контейнера** — верхняя граница плотности:
   без неё очень широкий монитор дал бы 7-8+ мелких колонок вместо
   разумных 4-6. Тот же приём уже применён на лендинге
   (`apps/web/public/landing.html`, `.page`/`.content`).

## Backend
**Node.js + Fastify + TypeScript**
Вместо Express: нативная валидация схем и типизация из коробки (через Zod),
меньше ручного wiring, выше производительность.

### Пакетная заливка каталога (Спринт 18)

Решено 20.07.2026, при планировании Спринта 18 (см. [TASKS.md](TASKS.md)).
Альтернатива ручному заведению карточек по одной
(`ProductForm.tsx`, Спринт 11) при большом каталоге, уже оцифрованном у
продавца в таблице.

- **Формат шаблона** — плоская таблица (`.xlsx`), одна строка = один
  вариант. Название и описание карточки повторяются на каждой строке
  варианта той же карточки (не только в первой) — устойчиво к
  сортировке/фильтрации строк в Excel, в отличие от схемы «описание
  только в первой строке группы». Отдельный лист на карточку
  рассматривался и отклонён: при 20-30 карточках (лимит Тарифа 1, см.
  [CONCEPT.md#тарифы](CONCEPT.md#тарифы)) — 20-30 вкладок неудобно
  листать и заполнять. Цены в шаблоне — в рублях (как и на форме
  `ProductForm.tsx`), конвертация в копейки на бэке через
  `lib/money.ts`-эквивалент.
- **Библиотека парсинга** — `exceljs`. Не `xlsx` (SheetJS): у
  community-редакции `xlsx` есть известные security-advisories
  (prototype pollution/ReDoS), закрытые только в платной Pro-версии —
  для эндпоинта, принимающего файлы от посторонних продавцов, это
  конкретный риск, не абстрактный. `exceljs` — MIT, активно
  поддерживается, потоковое чтение не тянет весь файл в память сразу.
- **Путь загрузки** — отдельный эндпоинт `POST
  /seller/products/import`, не переиспользование одиночного `POST
  /seller/products` в цикле — нужны транзакционность на уровне строки и
  построчный отчёт.
- **Партиальный импорт** — валидные карточки создаются, невалидные
  строки идут в построчный отчёт (`{row, error}`), а не блокируют весь
  файл целиком. Продавец с каталогом в 20 позиций и одной опечаткой
  получает 18 заведённых карточек и понятный список из 2 ошибок
  («строка 5: цена должна быть больше 0»), а не пустой результат с
  требованием переисправить файл до идеала. Валидация — той же
  Zod-схемой, что и ручной API (`createProductRequestSchema`,
  `packages/shared/src/schemas/products.ts`), не дублируется в парсере.
  Создание карточек — тот же сервис-слой `services/products.service.ts`
  (`createProduct`/`addVariant`), лимиты 30 карточек/10 вариантов (см.
  Спринт 11) проверяются там же.
- **Вне скоупа** — фото (пайплайн Спринта 16, добавляются через UI
  после импорта), скачивание пустого шаблона — статический `.xlsx` в
  `apps/web/public/` (тот же паттерн, что `offer.html`).

## Database
**PostgreSQL + Drizzle ORM**
Вместо Prisma: не тащит отдельный query-engine бинарник (легче образ,
быстрее холодный старт), query builder прозрачно мапится на SQL — не
прячет от разработчика, что реально уходит в базу, миграции — читаемые
`.sql`-файлы, а не сгенерированная магия.
Минус: меньше комьюнити и туториалов, чем у Prisma — будем разбираться
через документацию Drizzle/Postgres напрямую, если что-то пойдёт не так.

### Доменная схема (v1)

Согласована 18.07.2026. Восемь доменных таблиц (взамен заглушки
`health_check`):

- `sellers` — продавец: `telegram_id` (unique), `telegram_username`
  (обязателен — диплинк для связи с покупателем, см.
  `CONCEPT.md#коммуникация-продавца-и-покупателя`), ФИО и телефон
  (идентификация, см.
  `CONCEPT.md#модерация-и-лимиты`), название/описание магазина,
  `payment_details` (реквизиты для перевода на Тарифе 1), статус
  `active | blocked` (`blocked` — действие админа; скрытие витрины за
  неоплату — производное от статуса подписки, здесь не дублируется).
- `subscriptions` — подписка, 1:1 к продавцу в v1: тариф, `paid_until`,
  статус `active | grace | suspended | canceled`,
  `yk_payment_method_id` (токен карты для авторекуррента).
- `subscription_payments` — история списаний: сумма, статус,
  `yk_payment_id` (unique — идемпотентность вебхука ЮKassa на уровне
  констрейнта).
- `products` — карточка товара: название, описание, позиция сортировки,
  статус `active | hidden`.
- `product_variants` — вариант (размер/цвет/комплектация): название,
  `price_kopecks`, `old_price_kopecks` (nullable — зачёркнутая цена =
  скидка), `stock` (nullable — остаток; NULL = учёт остатка выключен),
  позиция (см. `CONCEPT.md#каталог-и-заказы`).
- `product_images` — фото карточки: `s3_key`, позиция.
- `orders` — заказ: продавец, `buyer_telegram_id`, статус, ПДн
  покупателя снапшотом (ФИО/телефон/адрес/комментарий), `consent_at`
  (факт согласия на обработку ПДн), `total_kopecks`.
- `order_items` — позиция заказа: ссылка на вариант (nullable,
  `SET NULL` при удалении варианта), снапшот названия товара/варианта и
  цены на момент заказа, количество.

Ключевые решения:

- **Без таблицы покупателей** — ПДн покупателя живут только снапшотом в
  заказе: минимизация по 152-ФЗ (профиль покупателя не строится),
  запрос на удаление данных = правка строк конкретных заказов.
- **Снапшоты в `order_items`** — правка/удаление товара продавцом не
  ломает историю заказов.
- **Деньги — integer в копейках** во всех таблицах.
- **Лимиты тарифа (карточки, варианты) — в сервис-слое**, не в
  CHECK-констрейнтах: лимит зависит от тарифа продавца.
- **Админы — список Telegram ID в env**, не таблица: один человек,
  таблица — «про запас».
- **Корзина — только на клиенте** (см. «Серверное состояние»), серверной
  таблицы корзины нет — заказ создаётся сразу оформленным.

### Бэкапы

Nightly `pg_dump` по cron/systemd timer на VM → загрузка в S3 (тот же
бакет, что и файлы товаров, папка `backups/`) — не отдельный managed-сервис
БД, а расширение уже принятой self-hosted модели (см. «Хостинг и деплой»).
Ретеншн: 7 дневных копий + 4 недельных. Отдельный weekly job разворачивает
последний дамп в чистую БД и проверяет успешность — бэкап, который ни разу
не восстанавливали, нельзя считать рабочим.
Минус: точка восстановления — раз в сутки (до 24ч данных теряется при
падении), а не PITR на любую секунду, как у Yandex Managed PostgreSQL.
Осознанный компромисс на старте (1-10 продавцов, некритичная потеря
данных за день); переезд на managed PostgreSQL — кандидат на пересмотр
при реальном росте нагрузки/критичности.

## Авторизация
Telegram `initData` проверяется на бэке (HMAC-подпись секретом бота) один
раз при открытии ТМА, дальше — своя короткоживущая сессия (JWT), а не
повторная проверка `initData` на каждый запрос.
Роль (покупатель/продавец/админ, см.
`CONCEPT.md#терминология`) не выбирается пользователем, а
резолвится на бэке по Telegram ID из `initData`: есть привязанная запись
продавца → доступна продавцовская админка, ID есть в списке админов →
доступна платформенная админка. Отдельного логина
(email/пароль) нет — см. `CONCEPT.md#интерфейсы-платформы`.

Реализация (`POST /auth`, 18.07.2026):

- **JWT — `@fastify/jwt`**, не `jsonwebtoken` вручную: типизированная
  интеграция в Fastify (подпись/проверка на инстансе) — тот же аргумент,
  что у axios/grammY. Секрет — `JWT_SECRET` из env, fail-fast на старте,
  если не задан.
- **Один короткоживущий токен (TTL 1 час), без refresh-токенов**: Mini
  App при каждом открытии получает свежий `initData` — протухший JWT
  фронт молча обменивает на новый через повторный `/auth`. Refresh-пара
  — лишняя машинерия, когда точка входа всегда под рукой.
- **Не «одна роль», а два независимых флага** в JWT и ответе `/auth`:
  `sellerId` (null — активного продавца нет) и `isAdmin`. Владелец
  платформы — одна учётка с обоими флагами. Контракт —
  `authRequest/ResponseSchema` в `packages/shared`.
- **Заблокированный продавец** (`status=blocked`) не получает `sellerId`
  уже при выдаче токена. **Механизм отзыва сессии решён и реализован в
  Спринте 14** (20.07.2026, платформенная админка): помимо этого,
  `requireSellerId` (`apps/api/src/auth/access.ts`, общий хелпер для всех
  `/seller/*` роутов — раньше дублировался по файлам, вынесен туда же) на
  каждый запрос перепроверяет статус продавца по БД, а не только наличие
  `sellerId` в JWT — заблокированный админом продавец теряет доступ к
  своей админке немедленно, не дожидаясь истечения TTL. Лишний DB-хит на
  запрос — цена, приемлемая на масштабе v1. `requireAdmin` рядом не
  перепроверяет `isAdmin` по БД — список админов живёт в env
  (`ADMIN_TELEGRAM_IDS`), не в таблице, перепроверять нечего.
- **`initData` старше 24 часов отвергается** — ограничение окна replay
  перехваченной строки.
- **`@fastify/rate-limit`** подключён с `global: false`, лимит висит
  точечно на `/auth` (20/мин, переопределяется `AUTH_RATE_LIMIT_MAX`) —
  пункт из ревью «сразу, а не после первого абьюза».
- Ошибки проверки — единый 401 без деталей (что именно не сошлось — не
  подсказывается); ПДн из `initData` не логируются (152-ФЗ).
- Список админов — `ADMIN_TELEGRAM_IDS` (числовые ID через запятую) в
  env, не таблица (см. «Доменная схема», ключевые решения).

Dev-режим (`AUTH_DEV_MODE=true`, 18.07.2026): вне Telegram (браузер на
локалке) настоящего `initData` нет — фронт dev-сборки шлёт
mock-`initData` (поля `user`/`auth_date`, но без валидной HMAC-подписи:
токен бота на фронт не попадает, подделать подпись нечем), а `/auth`
принимает его без проверки подписи, доверяя `user.id`. Иначе приложение
отлаживается исключительно внутри Telegram-клиента (см. «Локальный
дизайн-цикл и dev-бот»). Два предохранителя, потому что это единственное
место в бэке, где подпись `initData` не проверяется (в проде mock без
подписи = вход под любым Telegram ID):

- Флаг мёртв при `NODE_ENV=production` — исключён кодом, а не «не
  забудут выставить».
- `buildApp()` падает на старте, если флаг выставлен в проде (fail-fast,
  без override — в отличие от seed-guard'а: dev-байпас проверки подписи
  в проде не бывает осознанно легитимным).

Логика изолирована в `apps/api/src/auth/dev-mode.ts` (не подмешана
опцией в security-критичный `verifyInitData`).

Фронт (19.07.2026):

- **`initData` — тонкая обёртка над `window.Telegram.WebApp`**
  (`apps/web/src/lib/telegram.ts`), не тяжёлый SDK: авторизации нужны
  только сырой `initData` и `start_param`, а прямой доступ к `window`
  тривиально мокается. `resolveInitData()` отдаёт реальный `initData`, а
  в браузере вне Telegram (dev) синтезирует mock — потребитель
  `AUTH_DEV_MODE` выше. Скрипт `telegram-web-app.js` подключён в
  `index.html`.
- **Токен — в памяти модуля** (`auth/token-store.ts`), не в localStorage:
  живёт 1ч и переполучается из `initData` при каждом открытии, персистить
  незачем и небезопасно.
- **axios-интерцепторы** (`lib/api-client.ts`): request подставляет
  `Authorization: Bearer`, response на 401 разово пере-авторизуется
  (свежий `initData` → новый токен) и повторяет запрос; сам `/auth` и
  повторы исключены, чтобы не зациклиться.
- **`AuthProvider`** бутстрапит сессию до рендера маршрутов
  (loading/error-экраны), раздаёт её через `useSession`/`SessionContext`.
  Это app-lifecycle, а не кэшируемое серверное состояние, поэтому
  `useEffect`, а не TanStack Query (тот — для доменных данных).
- **Гварды и лендинг** по способностям сессии — см. «Роутинг».

## Роутинг
Одно SPA (`apps/web`), три группы маршрутов — витрина, продавцовская
админка, платформенная админка — разделены на уровне React Router,
доступность каждой группы определяется ролью из сессии (см.
«Авторизация»), а не отдельными сборками/деплоями. Общий UI-кит
переиспользуется между витриной и админками.

Вход в приложение: `main.tsx` читает `initData`, бэкенд проверяет HMAC
и выдаёт сессию с ролью (см. «Авторизация»). Если в `start_param`
пришёл `seller_id` — переход на витрину этого продавца; иначе продавец
попадает в свою админку, админ — в платформенную, покупатель без
параметра — на экран-заглушку («магазин открывается по ссылке
продавца»).

Карта экранов v1: витрина — каталог продавца, карточка товара, корзина,
чекаут, экран успеха (реквизиты продавца + кнопка в его личку + плашка
об ограничении ответственности); продавцовская админка — регистрация и
подписка, товары (CRUD), заказы, настройки (реквизиты, контакт);
платформенная админка — список продавцов, статус подписки, блокировка.

## Telegram-бот
**grammY** — клиент Bot API для платформенного бота: уведомления
продавцу (грейс-период подписки, блокировка, см. «Оплата подписки
продавцом» в `CONCEPT.md`) и будущая кнопка «Открыть магазин» через
администрирование бота в чужом канале (см.
`CONCEPT.md#кнопка-открытия-магазина`, вне скоупа v1).
Выбран вместо Telegraf: типизация Bot API «из коробки», совпадающая с
реальной схемой, без ручной подгонки — у Telegraf v4 типы разрослись и
стали сложнее в использовании, чем у нетипизированной версии (это
признаёт даже официальное сравнение на сайте grammY, независимо
подтверждается сторонними источниками). Плюс готовый webhook-адаптер
под Fastify (`webhookCallback(bot, "fastify")`) — не нужно руками
парсить и валидировать входящие апдейты.
Вместо **node-telegram-bot-api** — коллбэк-стиль, не TypeScript-first,
менее активно поддерживается.
Минус: полноценный bot-фреймворк ради узкого сценария (отправка
сообщений/кнопок и пара типов входящих апдейтов вроде
`my_chat_member` при выдаче прав администратора) — из middleware/session
API реально используется малая часть. Оправдано тем же аргументом, что
и выбор axios (см. «HTTP-клиент»): typed-клиент и webhook-роутинг не
пишутся вручную, а идут из коробки.

## Фоновые задачи
**pg-boss** (очередь задач поверх PostgreSQL) вместо отдельного брокера
(Redis/BullMQ) — используется для проверки грейс-периода подписки,

## Платежи и логистика (Тарифы 2-3)

**ЮKassa** (платежи) и **СДЭК** (логистика) — единственные интегрируемые
провайдеры на старте, продавец подключает их своими API-ключами

второй провайдер в каждой категории добавляется только при реальной боли,

Рассмотренные и отклонённые на старте альтернативы (справочно, на случай
если ЮKassa/СДЭК не подойдут — например, отказ в онбординге самозанятому
продавцу или нехватка ПВЗ СДЭК в его регионе):

- **Платежи**: Robokassa (агрегатор 50+ способов оплаты, автоотправка чека

  (отдельные условия для самозанятых).
- **Логистика**: Boxberry (тот же класс REST API, что и СДЭК, но меньше

  дней против 14; добавлять вторым, если у продавца в регионе гуще сеть

  слабее в федеральном покрытии — не замена СДЭК, а другой кейс), Почта
  России (самое широкое покрытие для дальних регионов, но заметно менее
  удобная интеграция/API).

## Хранилище файлов
**S3-совместимое объектное хранилище у российского провайдера**
(Yandex Object Storage / Selectel S3) через стандартный
`@aws-sdk/client-s3` — протокол одинаковый у всех S3-совместимых
провайдеров, при необходимости провайдера можно сменить без переписывания

Используется для карточек товаров, баннеров и кастомного фона витрины
(Тариф 3).
Не self-hosted (MinIO и т.п.): managed-хранилище снимает с команды бэкапы,
диски и масштабирование — на этапе запуска это не та боль, которую стоит
решать самостоятельно. Провайдер — российский, в одной юрисдикции с
остальными интеграциями (ЮKassa, СДЭК), а не AWS напрямую.

### Пайплайн фото товара (Спринт 16)

Зафиксировано 20.07.2026. Один слот фото на карточку (не на вариант, не
галерея) — минимальный достаточный скоуп v1, расширение до нескольких
фото — отдельная будущая задача при реальной потребности.

- **Загрузка** — прямой multipart через API (`@fastify/multipart`), не
  presigned URL: тот же принцип простоты, что и у остального API v1,
  лишний прыжок клиент↔S3 не оправдан объёмом трафика на масштабе v1.
  `POST /seller/products/:id/image`, `DELETE /seller/products/:id/image` —
  тот же паттерн ownership, что и у остального CRUD карточек (чужая
  карточка → 404, не 403).
- **Валидация**: `@fastify/multipart` режет файл лимитом размера (8 МБ) до
  того, как он весь окажется в памяти; MIME — только `image/jpeg`,
  `image/png`, `image/webp`. Защита от decompression bomb — `sharp`
  получает `limitInputPixels` (50 млн пикселей, с запасом покрывает
  реальные фото с телефона, но отсекает файл с честным малым весом и
  сфабрикованными огромными исходными размерами) — проверка идёт **до**
  полного декодирования, не после.
- **Ресайз** (`sharp`, конвертация в `webp` для обоих файлов): оригинал —
  большая сторона ≤1600px, отдельный `thumbnail` — большая сторона
  ≤400px (списки карточек грузят маленький файл, не полноразмерный).
- **Хранение**: та же таблица `product_images` (см. «Доменная схема
  (v1)» — заведена ещё в Спринте 9 вместе со всей доменной схемой,
  просто не использовалась до сих пор) — `s3_key` хранит ключ
  **оригинала**, ключ thumbnail — тот же путь с суффиксом `-thumb`
  перед расширением (не отдельная колонка, чтобы не тянуть лишнюю
  миграцию ради производного значения). В v1 на карточку — не больше
  одной строки; лимит проверяется в сервис-слое при загрузке (тот же
  принцип, что у лимитов карточек/вариантов, см. Спринт 11), не в
  БД-констрейнте — повторная загрузка удаляет старые объекты из S3 и
  заменяет строку, а не копит их. Путь в S3 — тот же бакет, что и у
  бэкапов, отдельный префикс `products/{sellerId}/{productId}/...`.
- **Отдача**: бакет приватный (см. запись Спринта 8 — «доступ только с
  авторизацией, не публичный»), поэтому фронт получает не голый URL
  объекта, а presigned GET-ссылку (`@aws-sdk/s3-request-presigner`, TTL
  1 час) — генерируется на лету при ответе `/shop/:sellerId` и
  `/seller/products`, вместе с обычными полями карточки. Альтернатива
  (публичный read-доступ на префикс `products/`) отклонена — она мешала
  бы в один бакет с приватными бэкапами разные уровни доступа, отдельный
  бакет под публичные фото ради этого заводить не стали, presigned-ссылка
  проще и не меняет модель доступа к бакету вовсе.

## Валидация
Zod-схемы в общем пакете (`packages/shared`), шарятся между backend-роутами
и frontend-формами — валидация не дублируется.

## HTTP-клиент
**axios** — и на фронте (запросы к `apps/api`), и на бэке (внешние
интеграции: ЮKassa, СДЭК). Интерцепторы (подстановка JWT, единая
обработка 401/ре-логин), единообразный формат ошибок (`error.response`)
и автоматическая (де)сериализация JSON удобнее писать один раз через
готовый API, чем поддерживать самописную обёртку над `fetch`.
Минус: отдельная зависимость и свой HTTP-движок поверх встроенного в
Node/браузер `fetch` — осознанный компромисс ради готовых интерцепторов.

## Серверное состояние (frontend)
**TanStack Query** поверх axios-клиента: кэш серверных данных,
инвалидация после мутаций, loading/error-состояния — во всех трёх
интерфейсах, вместо самописных хуков. Единственная state-библиотека в
проекте: корзина — клиентское состояние на React context + reducer, без
Zustand/Redux (один стор не оправдывает зависимость) и без серверной
корзины — заказ создаётся сразу оформленным, корзина теряется при
закрытии ТМА (принятый компромисс v1).

## Монорепо
pnpm workspaces:
```
apps/web/        — React
apps/api/        — Fastify
packages/shared/ — Zod-схемы, общие типы
```

`packages/shared` собирается через `tsc` (`main`/`types` смотрят на `dist/`,
не на `src/` напрямую) — обычный `node dist/index.js` в `apps/api` не умеет
исполнять TypeScript, поэтому пакетам, которые реально импортируют
`@grammashop/shared` в рантайме, нужен собранный `dist`. `pnpm -r
build`/`typecheck` собирают воркспейсы в топологическом порядке (`shared`
раньше зависящих от него пакетов) — pnpm делает это сам, без доп. настройки.
В Docker-сборке `apps/api` (см. `apps/api/Dockerfile`) `shared` собирается
явным отдельным шагом перед `api`.

## Docker
`docker-compose.yml` для локальной разработки (postgres + api + web),
multi-stage `Dockerfile` на каждый сервис для прода.

## Локальный дизайн-цикл и dev-бот

Итерации по дизайну и вёрстке идут локально, не через деплой на прод:
Vite dev-server с hot reload на хосте (`pnpm --filter @grammashop/web
dev`), браузер с мобильным вьюпортом в DevTools, `window.Telegram.WebApp`
мокается (см. dev-режим авторизации в «Авторизация»). Прод в дизайн-цикле
не участвует — на него смотрим только финальной проверкой перед закрытием
задачи: деплой каждой правки отступов — это минуты на итерацию вместо
секунд и мусорные коммиты в `main`.

Vite dev-server — на хосте, а не в контейнере: bind-mount репозитория в
докер ради file-watching на WSL2 — известный источник медленного/битого
HMR, а host-процесс даёт нативный watch. `postgres` + `api` при этом
поднимаются в докере (`docker compose up -d postgres api`; веб-контейнер
не нужен — его порт 5173 занимает host-Vite). Dev-server проксирует
`/api` на `api` (`server.proxy` в `vite.config.ts`), same-origin как в
проде (Caddy `handle_path /api/*`, см. `Caddyfile`) — фронт ходит в API
через тот же origin и локально, и через туннель, без CORS и второго
туннеля.

Что в браузере не проверить — поведение внутри реального клиента
Telegram: safe areas, шапка, разворачивание/сворачивание `viewport`,
клавиатура поверх форм, темы из `themeParams` (см. «Вёрстка»). Mini App
открывается только по HTTPS, `localhost` туда не подставить. Для этого —
**отдельный dev-бот** (BotFather, не платформенный прод-бот) +
**туннель с локалки**: `cloudflared tunnel --url http://localhost:5173`
(quick-туннель — бесплатно, без регистрации и аккаунта; альтернатива —
ngrok) даёт HTTPS-урл на локальный dev-server, урл прописывается в Mini
App dev-бота. Поддомены quick-туннелей (`*.trycloudflare.com`, `*.ngrok*`)
внесены в `server.allowedHosts` (`vite.config.ts`) — иначе Vite отвергает
запрос с чужого Host. Весь цикл поднимается одним скриптом
`scripts/dev-tunnel.sh` (докер + host-Vite + туннель, Ctrl-C гасит
Vite/туннель). Итерации локально с hot reload, телефоном открывается
dev-бот — то же приложение внутри настоящего Telegram.

Авторизация в этом цикле: dev-бот присылает *настоящий* `initData`,
подписанный *своим* токеном, которого у бэка нет. Поэтому `api`
поднимается с `AUTH_DEV_MODE=true` — `/auth` принимает подпись без
проверки (см. «Авторизация», «Dev-режим»); отдельно прописывать токен
dev-бота на бэке не нужно.

Минус: урл быстрого (quick) туннеля cloudflared меняется при каждом
перезапуске — его надо заново вбивать в BotFather. Терпимо, потому что
туннель нужен не постоянно, а на сеансы проверки Telegram-специфики;
если трение станет регулярным — именованный туннель cloudflared с
фиксированным доменом, это настройка, а не смена подхода.

## Хостинг и деплой
Один VM в Yandex Cloud (Compute Cloud) под `docker-compose` в проде — тот
же провайдер, что и для S3-хранилища (см. «Хранилище файлов»), меньше
разрозненных аккаунтов и биллингов ради одного небольшого проекта.
Старт — минимальная бюджетная конфигурация, взятая для отладки самого
деплоя (CI → ghcr.io → VM, Caddy/TLS), а не под реальную нагрузку;
апгрейд конфигурации — по факту трафика/продавцов, не заранее (см.
«Принципы проекта»).

Арендован 17.07.2026 (веха 7): `ru-central1-b`, платформа Intel Ice
Lake, 2 vCPU с гарантированной долей 20% (burstable — тестирование
прототипов без нагрузки, не прод-трафик), 2 ГБ RAM, 20 ГБ SSD, Ubuntu
24.04 LTS, статический публичный IP `89.169.191.173`. ~1700 ₽/мес.
Рабочий ориентир, не финальные параметры — апгрейд по факту
трафика/продавцов.

Без k8s/managed-оркестрации: `docker-compose` в проде уже выбран (см.
«Docker»), single-host модель этому соответствует, отдельный оркестратор
пока не нужен ни по нагрузке, ни по составу команды.
Минус: единая точка отказа, нет автомасштабирования — приемлемо для
старта, переезд на несколько инстансов за балансировщиком делаем позже,
когда появится реальный трафик, который это потребует.

Домен — **`grammashop.online`**, не `.ru`/`.рф`/`.su`: с 1 сентября 2026
управление этими зонами (включая смену DNS-записей) требует
идентификации администратора через Госуслуги — лишняя точка завязки на
гос-сервис без продуктовой выгоды, домен нигде не вводится пользователем
вручную (ссылка на магазин раздаётся через `startapp`, см.
`CONCEPT.md#кнопка-открытия-магазина`), национальная зона не даёт
преимущества в написании/узнаваемости. `grammashop.ru` куплен по
инерции до этого решения и не используется (оплачен до 17.07.2027,
автопродление выключено).

HTTPS обязателен: Telegram Mini Apps открываются только по HTTPS-урлу —
без него бот не откроется вообще, это не опция, а требование платформы.
Перед `api`/`web` в проде стоит **Caddy** как reverse proxy: сам
получает и обновляет сертификат Let's Encrypt (одна директива в
`Caddyfile`, без ручного certbot/cron), проксирует запросы на
`apps/api`, отдаёт статику `apps/web`. Выбран вместо nginx —
автоматический TLS из коробки экономит именно ту ручную возню
(генерация, обновление, редирект с 80 на 443), которую при nginx
пришлось бы настраивать отдельно; соответствует принципу "минимум
зависимостей про запас". Минус: меньше низкоуровневого контроля над
конфигурацией и меньше готовых рецептов в интернете, чем у nginx — при
текущем масштабе (один VM, один прокси перед двумя сервисами) это не
ограничение.

Роутинг в проде — единый origin, не поддомены: Caddy проксирует
`/api/*` (со strip-префиксом) на `apps/api`, всё остальное — на
`apps/web` (см. `docker/Caddyfile`). Убирает необходимость в CORS между
фронтом и бэком в проде (в отличие от локальной разработки, где `web` и
`api` — разные порты) и не требует второй A-записи под API-поддомен.
`apps/web` собирается в CI с `VITE_API_URL=/api` (build-arg, см.
`apps/web/Dockerfile`) — Vite подставляет значение в бандл на этапе
сборки, рантайм-конфигурация тут невозможна.

`docker-compose.yml` разнесён на три файла: база (общая для dev/прода),
`docker-compose.override.yml` (dev-only, подхватывается автоматически —
сборка образов из исходников, публикация портов на хост) и
`docker-compose.prod.yml` (образы из ghcr.io, без публикации портов
`api`/`web`/`postgres` — единственная точка входа снаружи это `caddy`
на 80/443).

## CI/CD
GitHub Actions (`.github/workflows/ci.yml`): прогон тестов (Vitest, см.
«Тестирование» — пайплайн падает, если тесты красные) → сборка
Docker-образов → пуш в registry → SSH на VM → `docker compose pull && up
-d` (джоба `deploy`, только на push в `main`, после успешной
`build-and-push`). Минимум тулинга, без отдельных CD-платформ (Argo,
Spinnaker и т.п.) — не нужны для одного VM.

Registry — **GitHub Container Registry (ghcr.io)**: авторизация через
встроенный `GITHUB_TOKEN`, без отдельных секретов/сервисных аккаунтов в
CI. Альтернатива (Yandex Container Registry, тот же облачный провайдер,
что и VM) отклонена — требует заводить сервисный аккаунт/ключ в Yandex
Cloud уже на вехе 6, ради шага, который всё равно нельзя проверить до
появления VM на вехе 7. Пакеты ghcr.io публичные (репозиторий
публичный) — VM пуллит образы анонимно, без `docker login`/PAT.

Деплой на VM — SSH под выделенным пользователем (не root), ключ
сгенерирован специально под этот доступ (не переиспользован от других
сервисов — компрометация одного не задевает остальные). Секреты
GitHub Actions: `VM_HOST` (статический IP, не домен — не зависит от
DNS), `VM_USER`, `VM_SSH_KEY` (приватный ключ, добавлен вручную через
веб-интерфейс GitHub — `gh` CLI не был авторизован в момент настройки).
Джоба `deploy`: `git pull --ff-only` в директории с клоном репозитория
на VM (репозиторий публичный — без токена), затем
`docker compose -f docker-compose.yml -f docker-compose.prod.yml pull &&
up -d`.

**`concurrency`-группа на весь workflow** (`cancel-in-progress: true` по
`${{ github.ref }}`) — добавлена 20.07.2026 после инцидента на Спринте
18: два пуша в `main` подряд запустили два параллельных рана,
`build-and-push` обоих пушит один и тот же мутируемый тег `:latest` —
более старый коммит финишировал на пару секунд позже и затёр `:latest`
свежего, `deploy` уже нового коммита стянул устаревший образ (SHA-теги
были в порядке, только `:latest` разъехался). Восстановление —
руками на VM, перетегированием образа по SHA обратно в `latest`
(см. историю инцидента в [TASKS.md](TASKS.md), Спринт 18). С группой
конкурентности более старый ран на той же ветке отменяется целиком при
новом пуше — до registry доезжает только последний коммит.

## Тестирование
Vitest — единый раннер для web и api. Компонентные тесты фронтенда —
`@testing-library/react` поверх того же Vitest (не отдельный раннер).

### Методология — TDD
Весь новый прикладной код (backend services, бизнес-логика фронтенда —
не разметка/стили) пишется циклом **red → green → refactor**: сначала
падающий тест на ожидаемое поведение, затем минимальный код, чтобы он
прошёл, затем рефакторинг без изменения поведения. Тест пишется до
реализации, а не после — иначе он рискует превратиться в фиксацию
того, что код уже делает, а не проверку того, что он должен делать.

Пирамида тестов:
- **Юнит-тесты** — основной объём. Services (бизнес-логика: лимиты
  тарифа, резолвинг роли, расчёт грейс-периода и т.д.) тестируются
  изолированно, без реальной БД.
- **Интеграционные тесты** — `route → service → тестовая БД` через
  встроенный `fastify.inject()` (без поднятия реального HTTP-сервера
  и сетевых вызовов) — на ключевые сквозные сценарии (авторизация,
  оплата, лимиты). Тестовая БД — отдельный сервис/схема Postgres в
  `docker-compose` (не мокаем Drizzle) — тесты видят тот же SQL, что
  и прод, а не поведение мока.
- **E2E не вводим на старте** — отдельный браузерный раннер
  (Playwright и т.п.) избыточен для текущего масштаба команды и
  продукта (см. «Принципы проекта»). Добавляем, если ручное сквозное
  QA перестанет справляться.

CI (см. «Хостинг и деплой» / «CI/CD») не пускает деплой дальше, если
тесты красные — это единственный автоматический гейт качества на
старте; отдельного шага линтера/type-check пока нет, добавляем при
реальной необходимости, а не заранее.

`apps/api` мигрирует `grammashop_test` автоматически перед прогоном
(vitest `globalSetup`, см. `apps/api/src/test/global-setup.ts`) — `pnpm
test` не полагается на то, что кто-то не забыл прогнать миграции руками
после `docker compose down -v`.

## Логирование и мониторинг
Встроенный логгер Fastify (`pino`) → stdout → `docker compose logs`, без
отдельного лог-агрегатора (ELK/Loki и т.п.) — на одном VM этого
достаточно, чтобы посмотреть логи руками при инциденте.
Sentry (free tier) — трекинг необработанных исключений на фронте и
бэке, единственное реальное дополнение к встроенному логированию: ловит
то, что легко пропустить при чтении логов вручную.
Минус: логи не централизованы и не ищутся полнотекстово — приемлемо для
старта, агрегатор добавляем, если это станет реальной болью.

## Принципы проекта
- Минимум зависимостей, ничего "про запас"
- Простая слоистая архитектура (routes → services → db), без DDD/hexagonal
- Усложняем архитектуру только тогда, когда появляется реальная боль,
  оправдывающая это — не заранее

## Пайплайн запроса
Не MVC (нет серверного View — фронтенд самостоятельный React SPA), а
слоистая схема `routes → services → db`, где у каждого слоя одна
ответственность:

1. **Frontend** — валидирует форму Zod-схемой из `packages/shared` перед
   отправкой (быстрая обратная связь пользователю), шлёт HTTP-запрос.
2. **Route (Fastify)** — тонкий слой: валидирует тело/параметры той же
   Zod-схемой из `packages/shared`, проверяет роль/сессию (см.
   «Авторизация»), вызывает нужный service. Без бизнес-логики.
3. **Service** — вся бизнес-логика (лимиты тарифа, принадлежность записи
   продавцу, доменные правила). Обращается к Drizzle напрямую — отдельного
   repository-слоя над Drizzle нет, это была бы абстракция ради
   абстракции при текущем масштабе.
4. **DB (Drizzle → Postgres)** — типобезопасные запросы, мультитенантность
   на уровне `WHERE seller_id = ...` внутри service-слоя, а не отдельная
   схема/база на продавца (см. `CONCEPT.md`).
5. Ответ идёт обратно тем же путём: service → route (сериализация и
   валидация по Zod-схеме ответа) → frontend получает данные, уже
   соответствующие контракту.

Когда `apps/api` разрастётся — `routes`/`services` нарезаются по доменам
(`products/`, `orders/`, `sellers/`) вместо одной плоской папки; сам
пайплайн (слои и их порядок) при этом не меняется.

Минус текущей схемы: service обращается к Drizzle напрямую, без
promise-слоя абстракции над доступом к данным — если понадобится
подменить способ доступа к данным, придётся менять вызовы в каждом
сервисе. Для текущего масштаба это осознанный компромисс.

## Скиллы Claude Code

Не часть рантайм-стека продукта — инструменты AI-ассистента для работы
над этим проектом, лежат в `.claude/skills/`. Часть активируется
автоматически по контексту диалога, часть — только по явной команде
(отмечено ниже).

### Дизайн и вёрстка (`apps/web`)

**`frontend-design`** (Anthropic, официальный) — активируется
автоматически при любой задаче на вёрстку/UI. Задаёт небанальные
дизайн-решения (типографика, палитра, лейаут) вместо шаблонного
"Inter + фиолетовый градиент + скруглённые карточки". Применять при
первой вёрстке витрины и обеих админок (см. «Роутинг», «UI-кит»).

**`web-design-guidelines`** (Vercel Labs) — 100+ правил
accessibility/UX/performance, роль строгого ревьюера после того, как
`frontend-design` предложил решение. Явно просить: «прогони через
web-design-guidelines» — после того как компонент/страница готовы, перед
тем как считать задачу закрытой.

### Безопасность (статический анализ, Trail of Bits)

Три скилла работают вместе, инструменты (`semgrep`/`codeql` CLI) нужно
ставить отдельно — скиллы учат Claude ими пользоваться, но не приносят
сами тулы.

- **`semgrep`** — быстрый scan по правилам (SQLi, XSS, небезопасные
  паттерны). Годится гонять регулярно, в т.ч. из CI.
- **`codeql`** — глубокий dataflow-анализ, дольше по времени. Для точечного
  аудита перед релизом или после изменений в авторизации/оплате.
- **`sarif-parsing`** — разбор и фильтрация результатов сканирования
  (общий формат SARIF, который отдают оба инструмента выше).

Применять: явно просить «прогони semgrep/codeql по `apps/api`» — в первую
очередь на модуле авторизации (`initData`/JWT, см. «Авторизация») и
интеграциях с ЮKassa/СДЭК, как только там появится код. До этого момента
скиллы неприменимы — сканировать нечего.

### Архитектура

**`senior-architect`** — не только советы, но и рабочие Python-скрипты:
- `architecture_diagram_generator.py` — диаграмма компонентов/слоёв/деплоя
  в Mermaid/PlantUML/ASCII из структуры проекта;
- `dependency_analyzer.py` — циклические зависимости, coupling score,
  устаревшие пакеты;
- `project_architect.py` — детектор паттерна (в нашем случае должен
  подтверждать слоистую схему `routes → services → db`, см. «Пайплайн
  запроса») и code smells (god-классы, смешение слоёв).

Применять периодически на растущем `apps/api` — сверять, что реальная
структура не расползлась и совпадает с зафиксированной здесь схемой, а
не заменять решения, уже принятые в этом документе (стек и пайплайн уже
выбраны — скилл не должен предлагать пересмотреть Fastify/Drizzle или
слоистую архитектуру на DDD, только проверять соответствие и чистоту
внутри неё).

### Docker

**`docker-development`** — три слэш-команды поверх `docker-compose.yml` и
`Dockerfile` (см. «Docker», «Хостинг и деплой»):
- `/docker:optimize` — размер образа, порядок слоёв, кеш сборки;
- `/docker:compose` — healthcheck'и, сети, volumes, dev/prod-профили;
- `/docker:security` — non-root user, `:latest`-теги, секреты в
  `ENV`/`ARG`, права контейнера.

Применять при написании `Dockerfile` для `apps/api`/`apps/web` и
`docker-compose.yml` — прогонять `/docker:security` обязательно перед
первым деплоем на прод VM (см. «Хостинг и деплой»), это единственный
security-гейт для контейнеров, аналогов в CI пока нет.

### Коммуникация

**`balanced`** — явный вызов `/balanced <тезис>` для структурированного
разбора решения (сильные стороны → жёсткая критика допущений →
альтернативы → синтез), отдельно есть `tldr`/`steelman`/`decision`-режимы.
Дефолтная прямота без вызова команды — не через скилл, а через раздел
«Стиль общения» в `CLAUDE.md`; `/balanced` нужен, когда именно нужен
формализованный многоходовой разбор, а не быстрый прямой ответ.
