### Спринт 31: подготовка к первому живому продавцу

Цель: покупательский путь чекаута устойчив к сетевым сбоям и гонке остатка
без риска задвоить заказ, сообщения бота продавцу/покупателю оформлены не
как голый служебный текст, у платформы есть растровый favicon на обеих
публичных поверхностях.

Состав согласован в диалоге 22.07.2026 (`/grill-me`). P0-блокеры v1 (ИП,
онбординг ЮKassa, юрпакет) — вне кода и вне этого спринта, идут своим
чередом параллельно. Юнит-экономика v1 — по теме, но не код, оставлена вне
спринта, отдельный заход.

Задачи:

- [x] Общая экспортируемая строка-константа в `packages/shared` для ошибки
  «нет остатка» — API (`orders.route.ts`/`orders.service.ts`) и фронт
  (`CheckoutPage.tsx`) сравнивают с одной и той же константой вместо
  дублирования литерала. Не заводить новое поле `code` в теле ошибки —
  во всём API сейчас единственный контракт `{ error: string }`, менять

- [x] Отдельное сообщение в `CheckoutPage.tsx` для сетевой/офлайн ошибки

  случая заходят в один общий `createOrder.isError`).
- [x] Idempotency-key на создание заказа: клиент генерирует UUID один раз

**Анализ перед стартом** (22.07.2026, перед командой старта):

- Idempotency-key (пункт 3) и точечные правки (пункты 1, 2, 6, 7) — состав
  чистый, паттерн стандартный (уникальный индекс + drizzle-миграция, тот
  же порядок, что и у существующих таблиц), ресёрч не нужен.
- Оформление бота (пункт 4, grammY) — проверил доку Bot API: `parse_mode:
  "MarkdownV2"` требует экранирования 18 спецсимволов вне сущностей
  (`_*[]()~\`>#+-=|{}.!`), а в `order-notification.ts` в текст
  интерполируются поля, которые вводит покупатель (ФИО, адрес,
  комментарий) — один неэкранированный символ ломает `sendMessage`
  (400 от Bot API) либо портит разметку. `parse_mode: "HTML"` требует
  экранировать только 3 символа (`&`, `<`, `>`), заметно безопаснее для
  интерполяции пользовательского текста. Меняю способ реализации в
  рамках той же задачи: HTML вместо MarkdownV2, с явным экранированием
  ПДн-полей перед подстановкой.
- Растровый favicon (пункт 5) — сверился с доступностью пакетов: `sharp`
  (уже зависимость `apps/api`) **не поддерживает вывод в `.ico`**
  нативно (подтверждено issue lovell/sharp#1118 — формата нет в libvips),
  вопреки формулировке задачи. Нужен компаньон для сборки контейнера ICO
  из PNG-буферов. Сравнил `to-ico` (последний релиз 2022, тяжёлые зависимости
  вроде `resize-img`/`parse-png`, дублирующие ресайз, который и так делает
  sharp) и `png-to-ico` (релиз 2026-07-06, лёгкие зависимости `pngjs`/
  `minimist`, принимает массив PNG-буферов напрямую — проверено по
  исходнику пакета) — беру `png-to-ico`. Sharp по-прежнему делает всю
  растеризацию из SVG и ресайз, `png-to-ico` только собирает `.ico`
  контейнер из готовых PNG-буферов 16/32/48. Способ реализации меняется
  в рамках той же задачи, скоуп не расширяется.
