### Спринт 43: Отмена заказа покупателем + напоминание о зависшем заказе

Цель: зависший/ненужный `new`-заказ закрыт с обеих сторон — покупатель
может сам погасить свой заказ, пока он `new`, а продавец получает
напоминание о заказе, висящем без ответа дольше 48 ч. Обе стороны
escape-hatch'а из грилла жизненного цикла достроены
([CONCEPT.md#жизненный-цикл-сущностей](../CONCEPT.md#жизненный-цикл-сущностей),
подразделы «Заказ» и «Покупатель»).

Состав согласован в диалоге 24.07.2026 (планирование после Спринта 42
через `/grill-me`). Взяты две связные задачи одной темы. Кап ширины
контента админок (Cart/Seller/Platform), булк-фото по имени файла,
категории, кадрирование — остаются в [backlog.md](backlog.md): гейтятся
отдельными решениями, не готовы к коду.

Сверка с живым кодом при планировании (по заметке о протухших
ревью-задачах): свипа по `new`-заказам нет, флага «напоминание отправлено»
на `orders` нет (`schema.ts`), паттерн daily-cron на pg-boss есть
(`SUBSCRIPTION_BILLING`/`SELLER_DELETION`, `queue/client.ts`); отмены
покупателем в [OrdersPage](../../apps/web/src/routes/storefront/OrdersPage.tsx)
нет (только статус + «Написать продавцу»), `updateOrderStatus`
seller-only, переход `new→canceled` и возврат остатка уже есть
(`orders.service.ts`). Оба кандидата подтверждены, не протухли.

## Анализ перед стартом

Выполнен 24.07.2026 первым шагом после команды старта. Обе задачи
подтверждены, состав не изменён — только уточнён способ реализации внутри
скоупа (extracted-хелпер отмены, бэкфилл-DML в миграции). Полный стоп не
требуется.

- **Задача 1, row-lock (флаг A).** Drizzle `.for("update")` уже используется
  в `createOrder` (лок строк вариантов) и в `updateOrderStatus` (лок вариантов
  при возврате остатка) — `orders.service.ts`. Внешнего ресёрча не нужно.
  Уточнение: `updateOrderStatus` берёт заказ через `findOwnedOrder` **вне**
  транзакции и не лочит саму строку заказа. Для покупательской отмены строку
  заказа надо лочить (`select … .for("update")` внутри tx) и перепроверять
  `status === 'new'` под локом — иначе гонка с продавцовским `new→paid`.
  Возврат остатка сейчас заинлайнен в транзакции `updateOrderStatus`
  (строки ~350–379) — вынести в общий `restoreOrderStock(tx, orderId)`,
  переиспользовать в обеих ветках (не дублировать).
- **Задача 2, pg-boss (флаг B).** Сверено с типами установленной версии
  `pg-boss@12.26.1`: `schedule(name, cron, data?, options?)`,
  `ScheduleOptions` c `tz` — сигнатура ровно та, что в
  `recurring-worker.ts`/`finalize-deletion-worker.ts`
  (`createQueue` → `schedule(queue, cron, null, { tz })` → `work`),
  регистрация в `index.ts` после `boss.start()`. Паттерн не менялся, беру
  его же. Текст напоминания — **не** `sendOrderNotification` (он тянет ПДн:
  ФИО/телефон/адрес), а отдельный форматтер поверх общих плюмбингов
  (`getBot`, `encodeOrderStartParam`, `MINI_APP_URL`, `InlineKeyboard`,
  `formatRubles`) — минимум без ПДн (152-ФЗ).
- **Бэкфилл миграции.** `drizzle-kit generate` даёт только DDL (nullable-
  колонка без дефолта). Бэкфилл `now()` текущим `new`-заказам — дописать
  `UPDATE` руками в сгенерированный `.sql` в той же миграции.

## Задача 1: Отмена заказа покупателем

Решение —
[CONCEPT.md#каталог-и-заказы](../CONCEPT.md#каталог-и-заказы) («покупатель
отменяет сам, пока заказ `new`») и
[CONCEPT.md#жизненный-цикл-сущностей](../CONCEPT.md#жизненный-цикл-сущностей),
подраздел «Покупатель» (представление «мои заказы» — обязательное, отмена
через него). Сейчас [OrdersPage](../../apps/web/src/routes/storefront/OrdersPage.tsx)
только показывает статус; отмена доступна лишь продавцу
(`PATCH /seller/orders/:id/status`).

- [x] Эндпоинт `POST /shop/:sellerId/orders/:id/cancel` (решение
  24.07.2026): per-shop, как `/orders/mine` (Спринт 40); авторизация по
  `request.user.telegramId` — владелец заказа, не продавец; чужой заказ →
  404 (не раскрывать существование). Фильтр выборки — по `id` +
  `sellerId` + `buyerTelegramId` вместе (`cancelOwnOrder`,
  `orders.service.ts`; роут — `orders.route.ts`).
- [x] **Атомарность (решение 24.07.2026):** отмена в транзакции — лок
  строки заказа (`.for("update")`), перепроверка `status === 'new'`
  внутри транзакции, затем `canceled` + возврат остатка. Нельзя отменить
  заказ, который продавец в ту же секунду пометил `paid`; отмена не из
  `new` → **409**.
- [x] Возврат остатка — **общий хелпер** `restoreOrderStock(tx, orderId)`,
  переиспользуемый и продавцовским `updateOrderStatus`, и покупательской
  отменой (транзакционная логика возврата не дублируется;
  `orders.service.ts`).
- [x] Фронт: кнопка «Отменить заказ» в `OrdersPage`, видимая только для
  заказов в статусе `new`, с confirm (отмена необратима); после успеха
  список рефетчится (инвалидация `["buyer-orders", sellerId]`), статус →
  «Отменён» (`useCancelOrder.ts`).
- [x] Тесты api (отмена из `new` → 200 + остаток вернулся; не из `new` →
  409; чужой заказ → 404; чужой `sellerId` → 404; гонка со сменой на
  `paid` не отменяет оплаченный) + web (кнопка только для `new`,
  confirm-гейт; `OrdersPage.test.tsx`); `pnpm --filter web build` — ок.

## Задача 2: Напоминание продавцу о зависшем `new` > 48 ч

Решение —
[CONCEPT.md#жизненный-цикл-сущностей](../CONCEPT.md#жизненный-цикл-сущностей),
подраздел «Заказ»: напоминание, если заказ висит в `new` дольше 48 ч,
тем же каналом (платформенный бот), **авто-отмены нет**. Вынесено из
Спринта 42 как отдельная подсистема.

- [x] Миграция: таймстемп `new_reminder_sent_at` (nullable) на `orders`
  — дедуп, чтобы не слать повторно (`schema.ts`,
  `drizzle/0007_overrated_matthew_murdock.sql`). **Бэкфилл (решение
  24.07.2026):** в той же миграции руками дописан `UPDATE … SET
  new_reminder_sent_at = now() WHERE status = 'new'` — фича работает
  только на будущие зависания, без залпа напоминаний по старым заказам на
  первом прогоне.
- [x] Третий cron-джоб на pg-boss (`order-reminder`), **раз в сутки**
  (решение 24.07.2026, 05:00 МСК) — по образцу `subscription-billing`/
  `seller-deletion-finalize` (`order-reminder-worker.ts`, регистрация в
  `index.ts`): свип находит заказы в `new` старше 48 ч с
  `new_reminder_sent_at IS NULL`, шлёт напоминание, ставит таймстемп.
  **Одно** напоминание (решение 24.07.2026). Свип вынесен в
  `runOrderReminderSweep()` — тестируемая единица отдельно от cron;
  флаг ставится только после успешной отправки (провал по одному заказу
  → Sentry + повтор следующим прогоном, не блокирует остальные).
- [x] Текст — **минимум без ПДн (решение 24.07.2026):** «Заказ #N … ждёт
  ответа больше 2 дней» + сумма + диплинк `o<orderId>` в админку; полные
  данные заказа — по диплинку. Отдельный форматтер
  `formatOrderReminderText` поверх переиспользованных плюмбингов
  (`getBot`, `encodeOrderStartParam`, `MINI_APP_URL`, `formatRubles` —
  экспортированы из `order-notification.ts`); **не** `sendOrderNotification`
  (он тянет ФИО/телефон/адрес). ПДн покупателя в напоминание не тянутся и
  не логируются (152-ФЗ).
- [x] Тесты api (свип отбирает только `new` старше порога и с пустым
  флагом; ставит флаг; не шлёт по моложе 48 ч; не шлёт повторно; заказ,
  ушедший из `new` в `paid`, не напоминается; при провале отправки флаг
  не ставится) + `pnpm --filter web build` — ок.
