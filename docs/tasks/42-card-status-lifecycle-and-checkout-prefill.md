Цель: продавец управляет тем, что на витрине (черновик → публикация →
снятие/архив, отдельно от удаления, включая пакетную публикацию

лишних карточек, покарточный тейкдаун админом, сквозной кабинет
покупателя, межмагазинный prefill) исключены — покрыты формулировками

Сверка с живым кодом при планировании (по заметке о протухших

## Анализ перед стартом

Проведён 24.07.2026 (первый шаг после команды старта). По каждой задаче
проверка разумности + ресёрч там, где есть выбор реализации.

- **Задача 1 (статус).** Диагноз спринта подтверждён живым кодом:
  `products.status` = `active | hidden`, DB-default `active`
  (`schema.ts:146`), read-side уже фильтрует `status='active'`
  (`shop.service.ts:47`), write-side его не выставляет — столбец мёртвый.
  Enum уже двухзначный в shared (`enums.ts:60`) — миграции enum не надо,
  только смена DB-default + проброс `status` в контракт. Last-variant
  guard сейчас безусловный (`products.service.ts:345`).
  **UX-ресёрч:** наша модель `active/hidden` = Shopify-паттерн (Active +
  Draft/Archived), где draft и archive осознанно схлопнуты в один
  `hidden`. Глагол «Опубликовать / Снять с витрины», статус — бейджем в
  списке. Подход совпадает с индустрией
  ([Shopify Product Status](https://litextension.com/blog/shopify-product-status/)).
- **Задача 2 (prefill).** Инфраструктура есть: снапшот ПДн в `orders`
  (`schema.ts:210`), чтение по `buyer_telegram_id + seller_id` уже в
  `listBuyerOrders` (`orders.service.ts:265`). **Нельзя** переиспользовать
  `/orders/mine` — он намеренно не отдаёт ПДн покупателя (`orders.ts:118`):
  нужен отдельный узкий эндпоинт «последний заказ для prefill».
- **Развилка, всплывшая при анализе → переформирование состава.** Импорт
  из Excel создаёт карточки через тот же `createProduct`
  (`products-import.service.ts:33`) — смена дефолта на `hidden` задевает и
  его, а CONCEPT про статус импорта молчал (гриль был про ручное
  создание). Решение пользователя 24.07.2026: импорт тоже рождает
  черновики (черновик-стадия осмыслена — фото не лезут в Excel,
  догружаются после), плюс действие «Опубликовать все черновики» — это
  **Задача 3** (расширение состава, зафиксировано в
  [CONCEPT.md#жизненный-цикл-сущностей](../CONCEPT.md#жизненный-цикл-сущностей)).
  Объём по фото — минимальный (решение пользователя): фото покарточно как
  сейчас + видимость черновиков в списке; булк-фото по имени файла → в
  [backlog.md](backlog.md). Полный стоп по регламенту отработан: состав
  переформирован в диалоге, получена новая команда старта, спринт выполнен.
