### Спринт 37: Удаление магазина с окном восстановления + retry Telegram-канала мониторинга

Цель: продавец (сам или через админа платформы) может удалить магазин без
безвозвратной потери данных — 30-дневное окно восстановления, по истечении
которого продавец обезличивается автоматически тем же способом, что сейчас
делается вручную; плюс независимый мелкий техдолг — повторная попытка
подключить Telegram-канал уведомлений Yandex Monitoring.

Состав согласован в диалоге 23.07.2026. Открытые вопросы задачи
«Удаление магазина продавцом/админом с окном восстановления» (бэклог,
взята в этот спринт) решены в диалоге:

- **Восстановить может** сам продавец из своего кабинета ИЛИ админ
  платформы — не только админ.
- **Срок окна** — 30 дней от момента удаления.
- **Подписка (`paid_until`)** при удалении приостанавливается, не
  сгорает — при восстановлении в пределах окна продавец получает обратно
  то же оставшееся оплаченное время.
- **Видимость витрины** — скрыта сразу, тот же паттерн, что у `blocked`
  (Спринт 32).
- **Причина удаления** — отдельное поле `delete_reason`, не переиспользуем
  `blocked_reason`: разные актёры и сценарии (самоудаление продавцом vs
  блокировка админом), общее поле смешало бы историю разных событий.
- **Финальное необратимое обезличивание** по истечении окна — новый
  ежедневный pg-boss job по образцу `recurring-worker.ts`/
  `runRecurringBilling` (Спринт 26), переиспользует логику текущего
  ручного `apps/api/src/db/anonymize-seller.ts` (вынести в сервис-функцию,
  вызываемую и из CLI-скрипта, и из воркера) — не отдельная копипаста
  обезличивания.
- **Уведомления** (продавцу о приближении конца окна; покупателям с
  незавершёнными заказами о судьбе заказа) — вне скоупа этого спринта,
  вынесены отдельным пунктом в бэклог по завершении.

## Анализ перед стартом

Состав не меняется, но чтение кода перед стартом вскрыло два места, где
план в тексте задачи выше был бы реализован неверно, если делать «в лоб»
— обе правки способа реализации внутри уже согласованного скоупа, без
стопа:

- **Экран восстановления живёт не в `SellerProfile.tsx`.** У
  `blocked`-продавца (Спринт 32) `sellerId` в сессии — всегда `null`
  (`resolveAuthContext`/`requireSellerId`, `auth.service.ts:50`,
  `apps/api/src/auth/access.ts:20-30` — фильтр `status = 'active'` на
  каждый запрос к `/seller/*`, не только при логине). У `deleted` та же
  механика сработает один в один — значит удалённый продавец физически
  не может открыть ни один `/seller/*`-роут, включая `SellerProfile`.
  Экран восстановления, как и `BlockedSeller`, должен жить на уровне
  `Landing.tsx` (новый `DeletedSeller.tsx` по образцу `BlockedSeller.tsx`)
  — рендерится по `session.sellerStatus === "deleted"`, до текущей
  развилки `sellerId`/`isAdmin`/`Fork`. Кнопка «Удалить магазин»
  (самоудаление, пока продавец ещё `active`) остаётся в
  `SellerProfile.tsx` — в этот момент `requireSellerId` ещё пропускает.
- **Самостоятельное восстановление не может идти через `requireSellerId`**
  — после удаления `sellerId` в сессии `null`, а значит и в
  `request.user.sellerId`, которым `requireSellerId` резолвит продавца.
  Нужен отдельный роут (по образцу `POST /seller/register`,
  `seller.route.ts:22-50`, который тоже не может использовать
  `requireSellerId` до того, как продавец существует) — резолвит
  продавца по `request.user.telegramId`, а не по `sellerId`, проверяет
  `status === 'deleted'` и окно (`deletedAt` + 30 дней), затем переводит
  в `active`. Админский путь восстановления по-прежнему штатно идёт через
  `PATCH /platform/sellers/:id/status` (`requireAdmin`, продавца знает по
  `:id` из URL, не по своей сессии).

Плюс один пробел, найденный по ходу чтения `billing.service.ts`, не
описанный в исходной формулировке задачи, но необходимый, чтобы решение
«подписка приостанавливается» действительно работало: `runRecurringBilling`
(`billing.service.ts:203-217`) выбирает `subscriptions` для списания без
`join` на `sellers` вообще — статус продавца не проверяется. Это значит,
что сегодня даже `blocked`-продавец продолжает списываться по рекурренту
(добавлено до Спринта 32, тем спринтом не тронуто и не замечено). Без
фикса «приостановка» для `deleted` была бы фикцией — ЮKassa продолжала бы
списывать карту закрытого магазина. Простейший и единственный
естественный фикс — добавить `join sellers` и фильтр `sellers.status =
'active'` в выборку `due`; он симметрично закрывает и старый пробел с
`blocked`, отдельного кода под `deleted` не требуется. Добавлено отдельным
пунктом ниже (не расширение цели спринта — необходимое условие уже
принятого решения о подписке).

pg-boss (`^12.26.1`) — новый воркер переиспользует уже работающий в
проекте паттерн `boss.schedule`/`boss.work` (`recurring-worker.ts`,
Спринт 26) один в один, отдельная сверка с документацией pg-boss не
нужна — версия и API те же, что уже проверены в бою.

Задачи:

**Довесок**: retry настройки Telegram-канала уведомлений Yandex
Monitoring (`@YandexCloudNotify_bot` не отвечал на `/start` — ни из чата
напрямую, ни по диплинку из консоли, см. Спринт 27) — на этот раз бот
ответил, канал «Телеграм мониторинг» заведён и привязан к обоим алертам
(`grammashop-health-ping`, `grammashop-backup-heartbeat`) рядом с email,
не вместо него; подтверждено живыми WARN-уведомлениями в Telegram
23.07.2026.

Заодно, по ходу проверки `grammashop-health-ping` (пользователь открыл
`grammashop.online/health` в браузере — редирект на лендинг вместо JSON),
найден и исправлен реальный баг, живший с самого Спринта 27: Caddy
проксирует в `apps/api` только `/api/*` (`docker/Caddyfile`), а
`HEALTH_URL` у Cloud Function `grammashop-health-check` был указан без
префикса — `https://grammashop.online/health` вместо
`https://grammashop.online/api/health`. Функция получала HTML лендинга
вместо JSON, `res.json()` падал, `checkHealth()` ловил исключение и писал
`health.ok = 0` при живом и полностью исправном бэкенде — алерт молча
врал все это время (сигнала не было, потому что единственный канал,
email, видимо, никто не проверял достаточно внимательно). Исправлено:
комментарий в [infra/yandex-health-check/index.js](../../infra/yandex-health-check/index.js)
(документирует правильный URL) + сама переменная окружения `HEALTH_URL`
в консоли Yandex Cloud Functions (не в репозитории — сама функция
деплоится отдельно от монорепо, см. комментарий в файле). После правки
алерт на глазах перешёл Alarm → Warning, дальше самостоятельно уйдёт в Ok
по мере заполнения окна вычисления хорошими точками.

## Обзор

Все задачи кода выполнены как описано выше, состав не менялся по ходу
работы (обе правки способа реализации и билинг-фикс — см. «Анализ перед
стартом» — были учтены сразу, до кода, полного стопа не требовалось).
Заодно, помимо основной цели, изменение затронуло:
- `--catalog-max-width`/сетка каталога — не трогали, отдельный от этого
  спринта пересмотр (см. коммит «Пересмотр принципа сетки каталога»,
  тот же день, но отдельное решение и отдельная задача в бэклоге).
- Общий источник окна восстановления (`SELLER_RESTORE_WINDOW_DAYS`,
  `packages/shared/src/domain/seller-deletion.ts`) — вынесен в shared, а
  не задан двумя магическими «30» на бэке и фронте по отдельности.

`pnpm -r typecheck` и `pnpm -r test` — зелёные (23 файла/197 тестов
`apps/api`, 12 файлов/77 тестов `apps/web`, включая новые тесты на
самоудаление/восстановление, админский переход в `deleted`, пропуск
рекуррентного списания для `blocked`/`deleted` и экран `DeletedSeller`).

Не проверено вживую (то же ограничение среды, что и в прошлых спринтах,
см. Спринты 33-36): реальный клик по кнопкам «Удалить»/«Восстановить» в
Telegram Desktop/мобильном клиенте — нужна проверка пользователем.
Довесок про Yandex Monitoring выполнен и подтверждён пользователем
целиком, включая попутный фикс `HEALTH_URL` (см. выше).
