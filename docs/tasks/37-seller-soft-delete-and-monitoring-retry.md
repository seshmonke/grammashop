### Спринт 37: Удаление магазина с окном восстановления + retry Telegram-канала мониторинга

Цель: продавец (сам или через админа платформы) может удалить магазин без
безвозвратной потери данных — 30-дневное окно восстановления, по истечении
которого продавец обезличивается автоматически тем же способом, что сейчас
делается вручную; плюс независимый мелкий техдолг — повторная попытка
подключить Telegram-канал уведомлений Yandex Monitoring.

Состав согласован в диалоге 23.07.2026. Открытые вопросы задачи из
[backlog.md](backlog.md#удаление-магазина-продавцоманом-с-окном-восстановления)
решены там же:

- **Восстановить может** сам продавец из своего кабинета ИЛИ админ
  платформы — не только админ.
- **Срок окна** — 30 дней от момента удаления.
- **Подписка (`paid_until`)** при удалении приостанавливается, не
  сгорает — при восстановлении в пределах окна продавец получает обратно
  то же оставшееся оплаченное время.
- **Видимость витрины** — скрыта сразу, тот же паттерн, что у `blocked`
  (Спринт 32).
- **Причина удаления** — отдельное поле `delete_reason`, не переиспользуем
  `blocked_reason`: разные актёры и сценарии (самоудаление продавцом vs
  блокировка админом), общее поле смешало бы историю разных событий.
- **Финальное необратимое обезличивание** по истечении окна — новый
  ежедневный pg-boss job по образцу `recurring-worker.ts`/
  `runRecurringBilling` (Спринт 26), переиспользует логику текущего
  ручного `apps/api/src/db/anonymize-seller.ts` (вынести в сервис-функцию,
  вызываемую и из CLI-скрипта, и из воркера) — не отдельная копипаста
  обезличивания.
- **Уведомления** (продавцу о приближении конца окна; покупателям с
  незавершёнными заказами о судьбе заказа) — вне скоупа этого спринта,
  вынесены отдельным пунктом в бэклог по завершении.

Задачи:

- [ ] Схема: `sellers.status` — третье значение enum'а, `deleted`
  (параллельно `blocked`, `packages/shared/src/domain/enums.ts` +
  `sellerStatuses`); новые поля `sellers.deleted_at` (timestamp,
  nullable), `sellers.delete_reason` (text, nullable) — миграция
  drizzle-kit.
- [ ] API: soft-delete эндпоинт для продавца (свой кабинет) — переводит
  текущий статус в `deleted`, требует причину (свободный текст, как у
  блокировки), проставляет `deletedAt`.
- [ ] API: восстановление — продавцом (в пределах 30 дней от
  `deletedAt`) и админом (расширить существующий
  `PATCH /platform/sellers/:id/status`, Спринт 32, третьим переходом) —
  возврат в `active`, очистка `deletedAt`/`deleteReason`.
- [ ] `resolveAuthContext`/`authResponseSchema` (`auth.service.ts`,
  Спринт 32) — расширить логику `blocked` (`sellerId: null`,
  `sellerStatus`/причина в ответе `/auth`) на `deleted`, плюс дата
  истечения окна для экрана в кабинете.
- [ ] UI кабинета продавца (`SellerProfile.tsx`) — кнопка «Удалить
  магазин» (подтверждение + причина); если магазин уже `deleted` и окно
  не истекло — экран с датой истечения и кнопкой «Восстановить» (по
  аналогии с `BlockedSeller`/`Landing.tsx`, Спринт 32).
- [ ] UI платформенной админки (`PlatformHome.tsx`) — расширить
  существующий UX блокировки третьим переходом в/из `deleted`.
- [ ] Новый pg-boss daily job (`finalize-seller-deletion`, по образцу
  `recurring-worker.ts`) — находит продавцов `status = 'deleted'` с
  `deletedAt` старше 30 дней, вызывает вынесенную сервис-функцию
  обезличивания, идемпотентен при повторном запуске.
- [ ] typecheck + `pnpm -r test` зелёные.

**Довесок**: retry настройки Telegram-канала уведомлений Yandex
Monitoring (`@YandexCloudNotify_bot` не отвечал на `/start`, см.
[backlog.md](backlog.md#telegram-канал-уведомлений-в-yandex-monitoring))
— если проблема всё ещё на стороне Yandex, оставить на email fallback ещё
раз, не блокировать спринт.

Реальная работа (код, коммиты) не начинается без отдельной команды старта.
