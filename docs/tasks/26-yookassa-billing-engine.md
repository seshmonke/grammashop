### Спринт 26: Биллинг-движок подписки на ЮKassa (sandbox)

Цель: интеграция с ЮKassa для оплаты подписки продавцом написана и покрыта
тестами против тестового магазина ЮKassa (sandbox) — к бою остаётся только
подмена ключей на боевые. Независимо от внешних P0-блокеров (ИП, юрпакет,
модерация мерчант-аккаунта) — они нужны только для выхода на бой, не для
разработки. Состав согласован в диалоге 21.07.2026 (сразу после закрытия
Спринта 25) — намеренно один пункт, без довесков, чтобы не размазывать
присест на крупной задаче.

Задачи:

- [x] Создание платежа + сохранение `payment_method_id`
  (`subscriptions.yk_payment_method_id`, схема уже готова).
  `POST /seller/subscription/pay` (`routes/billing.route.ts`) →
  `startSubscriptionPayment` (`services/billing.service.ts`): платёж с
  `save_payment_method` + `confirmation_url`, токен сохраняется по вебхуку
  `payment.succeeded` в `settlePayment`. Клиент к ЮKassa — свой на `fetch`
  (`yookassa/client.ts`), без SDK.
- [x] Джоба рекуррентного списания на pg-boss (расписание своё — у ЮKassa
  нет планировщика). Ежедневный cron-свип (`billing/recurring-worker.ts` +
  `runRecurringBilling`): списывает подписки с истёкшим `paid_until` и
  сохранённым токеном; после провала — grace/suspended по 3-дневному окну
  от `paid_until` (без новой колонки).
- [x] Вебхук-роут с идемпотентностью (повторная доставка события не должна
  дважды продлевать подписку) + rate limit сразу на этот роут через
  `config.rateLimit` (снято из Спринта 25 21.07.2026 именно потому, что
  роута ещё не было — теперь есть на чём вешать). `POST /billing/webhook`:
  телу не доверяет (у ЮKassa нет подписи) — перечитывает платёж
  `GET /payments/{id}`; идемпотентность двумя слоями: unique `yk_payment_id`
  + атомарный переход `pending → terminal` (условный UPDATE). IP-allowlist
  ЮKassa — defense-in-depth, включается на бою (`YOOKASSA_VERIFY_WEBHOOK_IP`).

#### Ретро

Все три задачи закрыты, скоуп не расширялся. Тесты: `billing.route.test.ts`
(pay + идемпотентность вебхука + canceled), `billing.service.test.ts`
(свип: продление, grace, suspended), `yookassa/client.test.ts` (контракт
запроса), `yookassa/webhook-ip.test.ts` (CIDR-матчинг) — 175/175 зелёные,
типы чистые. Схему не трогали — колонки были готовы, миграции нет.
Всплывшее в backlog: (1) UI-кнопка «оплатить подписку» в админке продавца
поверх нового эндпоинта; (2) бой требует включить рекуррент у менеджера
ЮKassa + `YOOKASSA_VERIFY_WEBHOOK_IP=true` + боевые ключи.

**Довесок 21.07.2026 — сквозная проверка против реального sandbox.**
Изначальные тесты гоняли клиент ЮKassa только мокнутым — цель спринта
дословно требовала «покрыта тестами против тестового магазина ЮKassa»,
живого похода в API не было. Закрыто: получен тестовый магазин в личном
кабинете ЮKassa (не требует ИП/модерации — не пересекается с блокером
P0 «Онбординг мерчанта»), добавлен ручной скрипт `verify:yookassa`
(`yookassa/verify.ts`, по образцу `s3/verify.ts`, не часть CI) — создаёт
платёж с `save_payment_method`, ждёт ручной оплаты тестовой картой, затем
сам пробует рекуррентное списание по сохранённому `payment_method_id` без
подтверждения. Прогнан вживую 21.07.2026: оба платежа succeeded, токен
сохранился и переиспользовался — ровно тот путь, что использует
`billing.service.ts` в проде. Ключи тестового магазина — в локальном
`.env` (`YOOKASSA_SHOP_ID`/`YOOKASSA_SECRET_KEY`, не коммитится).

#### Анализ перед стартом (21.07.2026)

Ресёрч (ЮKassa API, pg-boss v12, TS-SDK) ни одну задачу не опрокинул —
поменял только способ реализации внутри скоупа. Полного стопа нет:
провайдер (ЮKassa) в CONCEPT уже зафиксирован, ПДн за пределы РФ не идут
(российский процессор, карта токенизируется в `payment_method_id`).

- **Клиент к ЮKassa — свой на `fetch`, без SDK.** Единственный TS-SDK
  ([@webzaytsev/yookassa-ts-sdk](https://github.com/WEBzaytsev/yookassa-ts-sdk),
  6★, один мейнтейнер) в денежном пути — supply-chain риск ради двух
  вызовов. Basic auth `ShopID:SecretKey` + заголовок `Idempotence-Key`.
- **Рекуррент (задача 1–2)** ([автоплатежи](https://yookassa.ru/developers/payment-acceptance/scenario-extensions/recurring-payments/pay-with-saved)):
  первый платёж — `save_payment_method:true` + `confirmation_url`,
  `payment_method.id` сохраняем по `payment.succeeded`; списание —
  `POST /v3/payments` с `payment_method_id`, без подтверждения. Рекуррент
  включён по умолчанию только в тестовом магазине — это и есть скоуп
  (sandbox), бой = звонок менеджеру ЮKassa (шаг «к бою», не блокер).
- **Расписание — ежедневный cron-свип на pg-boss** (`schedule()`), не
  цепочка `sendAfter`: пропущенный запуск догоняется на следующий день,
  источник истины — `paid_until`, а не хрупкая цепочка отложенных job.
  Совпадает с концептовским «можно пропустить/отодвинуть, просто не
  создавая платёж».
- **Вебхук (задача 3)** ([Notifications](https://yookassa.ru/developers/using-api/webhooks)):
  у ЮKassa нет криптоподписи — тело `{type,event,object}` не доверяем,
  перечитываем платёж `GET /v3/payments/{id}` и продлеваем только по
  реальному статусу. IP-allowlist (185.71.76.0/27 и др.) — defense-in-depth
  за Caddy. Идемпотентность: unique `yk_payment_id` (в схеме) + перечитка.
  Возвращаем 200, иначе ЮKassa ретраит 24ч.
- **Сумма списания — параметр конфига.** Цена Premium не зафиксирована
  («считается позже»), Free-верификация — 1 ₽; в sandbox любая. Не блокер.
- **Скоуп-заметка:** первый платёж возвращает `confirmation_url` для
  открытия продавцом — бэкенд-эндпоинт (создать платёж + вернуть URL,
  захват метода по вебхуку) входит в задачу 1; кнопку в UI админки в
  скоуп не раздуваю — естественное продолжение.
