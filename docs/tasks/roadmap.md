## Дорожная карта проекта

Высший масштаб планирования: инженерный трек от пустого репозитория до
работающего деплоя на сервере. Это не про фичи продукта (подписки,
каталог, чекаут и т.д. — они появятся как задачи внутри веток «Бэкенд»/
«UX/UI» позже) и не про юридическо-организационные вещи (оферта,
юрлицо, пилотный запуск) — те заводятся отдельными задачами, когда до
них дойдёт очередь. Здесь — только базовые инструменты и порядок, в
котором они поднимаются, чтобы можно было работать на следующем уровне
абстракции. Веха 1 — фундамент; 2-4 — следующий слой, может идти
параллельно; 5-6 — финал. Редактируется по ходу работы.

- [x] **1. Инициализация проекта** — монорепо (pnpm workspaces), базовая
  структура `apps/web`/`apps/api`/`packages/shared`, `package.json` и
  TypeScript-конфиги в каждом пакете, `.env.example` и схема
  обязательных переменных окружения (БД, JWT-секрет, ключи S3 и т.д.),
  минимальный `.gitignore`.
- [x] **2. Докер и локальное окружение** — `docker-compose.yml`
  (postgres + тестовая БД/схема + api + web), multi-stage `Dockerfile`
  на каждый сервис, `docker compose up` поднимает всё локально и
  отдаёт "hello world" с api и web.
- [x] **3. База данных** — подключение Drizzle к Postgres, первая
  миграция, базовая схема (заглушка), проверка миграций в докере.
- [x] **4. Бэкенд** — скелет Fastify, пайплайн `routes → services → db`
  (см. `STACK.md#пайплайн-запроса`), Zod-схемы в `packages/shared`,
  первый health-check эндпоинт — по TDD (см. `STACK.md#тестирование`):
  сначала падающий тест через `fastify.inject()`, потом код.
- [x] **5. UX/UI** — скелет React + Vite, `shadcn/ui` подключён,
  роутинг трёх групп маршрутов (см. `STACK.md#роутинг`), axios
  настроен на обращение к `apps/api`, первый компонент — тоже по TDD
  (`@testing-library/react` + Vitest).
- [x] **6. CI/CD** — GitHub Actions: прогон тестов (Vitest, красные —
  пайплайн падает) → сборка Docker-образов → пуш в ghcr.io. SSH-деплой
  на VM отложен до вехи 7 (сервера ещё нет), см.
  `STACK.md#ci-cd`.
- [x] **7. Деплой на сервер** — VM в Yandex Cloud (старт — минимальная
  бюджетная конфигурация для отладки деплоя, апгрейд по факту нагрузки,
  см. [STACK.md#хостинг-и-деплой](STACK.md#хостинг-и-деплой)) ✅,
  `docker-compose` в проде с Caddy как reverse proxy (автоматический TLS
  через Let's Encrypt — обязателен, ТМА открывается только по HTTPS) ✅,
  S3-хранилище и Sentry подключены и проверены на бою ✅, nightly `pg_dump`
  в S3 настроен и еженедельная проверка восстановления работает (см.
  [STACK.md#бэкапы](STACK.md#бэкапы)) ✅.

## Дорожная карта продукта (v1)

Следующий уровень планирования после инженерного трека выше — фичи,
нужные для сквозного пути «продавец подписался → покупатель сделал
заказ» в объёме Тарифа 1 (см.
[CONCEPT.md#скоуп-v1-mvp](CONCEPT.md#скоуп-v1-mvp)). Пункты идут в
предполагаемом порядке зависимостей, порядок может уточняться по ходу —
это не скрам-доска на неделю вперёд, а следующий уровень дорожной карты
после инженерного трека (вехи 1-7 выше), который до этого момента нигде
не был явно разложен.

1. **Доменная схема** — таблицы Drizzle для продавцов, товаров с
   вариантами (карточка + варианты, цена на варианте, ≤10 вариантов на
   карточку — см.
   [CONCEPT.md#каталог-и-заказы](CONCEPT.md#каталог-и-заказы)) и заказов
   со статусной моделью «новый → оплачен → выполнен / отменён» (взамен
   схемы-заглушки `health_check`, см. веху 3). Схема согласована
   18.07.2026 →
   [STACK.md#доменная-схема-v1](STACK.md#доменная-схема-v1) — осталось
   написать миграции.
2. **Авторизация** — проверка `initData` (HMAC), выдача JWT-сессии,
   резолвинг роли (покупатель/продавец/админ) по Telegram ID (см.
   [STACK.md#авторизация](STACK.md#авторизация)).
3. **Регистрация и подписка продавца** — форма регистрации, интеграция
   ЮKassa (автоплатежи), грейс-период 3 дня, скрытие витрины при
   неоплате (см.
   [CONCEPT.md#оплата-подписки-продавцом](CONCEPT.md#оплата-подписки-продавцом)).
   Незаблокированная часть (форма регистрации без оплаты, скрытие
   витрины без подписки, льготы от админа) — в работе Спринтом 21;
   ЮKassa-интеграция по-прежнему ждёт внешние блокеры (ИП + модерация
   мерчанта + юрпакет).
   Сюда же встраивается идемпотентность вебхука ЮKassa (уже в очереди
   ниже) — не отдельный этап в конце. **Важное ограничение, вскрытое
   20.07.2026 при проверке уведомлений о заказах** (см. «Очередь»,
   «Уведомления продавцу через Telegram-бота» — сетевая проблема с
   прод-VM отдельно, это ограничение — про сам механизм, не про сеть):
   Telegram Bot API не даёт боту писать пользователю первым — только
   если пользователь сам хотя бы раз начал с ботом диалог/открыл его
   Mini App. Раз регистрация продавца в любом случае идёт через открытие
   `@grammashopbot`, это условие выполняется автоматически самим фактом
   регистрации — специально предусматривать ничего не нужно, но это
   именно предположение, которое стоит явно проверить при реализации
   формы регистрации (не полагаться молча), и учесть кейс «продавец
   заблокировал бота после регистрации» — `sendMessage` в этом случае
   вернёт 403, обрабатывается тем же путём, что и сетевые сбои (см.
   `notifications/order-notification.ts` — ошибка воркера не роняет
   покупательский флоу, видна как `failed` job в pg-boss).
4. **Продавцовская админка: товары** ✅ — CRUD карточек товара с лимитом
   30 шт. (Тариф 1, см. [CONCEPT.md#тарифы](CONCEPT.md#тарифы)) сделан
   Спринтом 11, проверено на бою с телефона. Загрузка изображений — вне
   этого пункта, в работе Спринтом 16.
5. **Витрина и каталог** ✅ — рендер витрины продавца по `seller_id` из
   `startapp`, список товаров, карточка товара — сделан Спринтом 9,
   проверено на бою с телефона.
6. **Корзина и чекаут** ✅ — сборка корзины, оформление заказа с полной
   формой (ФИО, телефон, адрес, комментарий) и согласием на обработку
   ПДн; на Тарифе 1 вместо формы оплаты картой — экран с
   реквизитами/контактом продавца (см.
   [CONCEPT.md#каталог-и-заказы](CONCEPT.md#каталог-и-заказы)) — сделано
   Спринтом 12, проверено на бою с телефона. Уведомление продавцу о
   новом заказе через платформенного бота и авто-списание остатков —
   тоже реализованы; списание остатков подтверждено на бою, доставка
   уведомления — нет (внешняя сетевая проблема прод-VM, не код, см.
   «Очередь»). Rate limiting на создание заказов встроен сразу, не после
   первого абьюза.
7. **Продавцовская админка: заказы** ✅ — список заказов, смена статуса
   вперёд (оплачен/выполнен) и откат ошибочного клика назад — сделано
   Спринтом 13, проверено на бою с телефона.
8. **Платформенная админка** ✅ — список продавцов, статус подписки,
   ручная блокировка/разблокировка — сделано Спринтом 14, проверено на
   бою с телефона (блокировка демо-продавца сразу скрывает витрину,
   разблокировка возвращает).

Все 8 пунктов дорожной карты продукта v1 реализованы и проверены на бою.
Единственное открытое ограничение — п.3 (регистрация и оплата подписки
продавца) технически написан лишь частично (авторизация — да, форма
регистрации и интеграция ЮKassa — нет): блокер внешний, не инженерный
(модерация мерчант-аккаунта ЮKassa + юрпакет 152-ФЗ, см. «Очередь»), но
именно он не даёт объявить v1 полностью готовым к внешней проверке — см.
критерий готовности ниже.

Критерий готовности v1 (внутренняя + внешняя проверка) — см.
[CONCEPT.md#скоуп-v1-mvp](CONCEPT.md#скоуп-v1-mvp).
