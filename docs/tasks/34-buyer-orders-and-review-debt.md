### Спринт 34: Заказы покупателя + технический долг из ревью 23.07

Цель: у покупателя появляется способ увидеть свои заказы (эндпоинт «мои
заказы» с авторизацией по Telegram ID + экран истории/деталей, третья
вкладка floating toolbar), а свежий технический долг из ревью
[2026-07-23.md](../reviews/2026-07-23.md) закрыт тем же заходом — фича и долг
здесь про один и тот же запрос (`orders` по `buyer_telegram_id`, которому
как раз нужен индекс).

Состав согласован в диалоге 23.07.2026 (по итогам ревью проекта). Команда
старта дана 23.07.2026.

## Анализ перед стартом

Все пять задач разумны и не расширяют скоуп — анализ чистый, работа
продолжена без стопа.

- **«Мои заказы» (бэк).** Эндпоинт сознательно не скопирован по образцу
  `/seller/orders` (фильтр по `sellerId`) — по формулировке задачи фильтр
  только по `buyerTelegramId`, т.е. список **сквозной по всем магазинам**
  платформы (один бот `@grammashopbot`, разные продавцы различаются
  `start_param`, покупатель мог заказывать у нескольких). Значит в ответе
  нужен идентификатор магазина на каждый заказ (`sellerId` + `shopName`,
  через join с `sellers`) — иначе плоский список нечитаем. Пагинация не
  нужна: `listSellerOrders` тоже без неё, и это согласуется с масштабом v1
  (один живой продавец, из ревью 23.07). Роут — `GET /orders/mine`, только
  `fastify.authenticate` (без `requireSellerId` — любой авторизованный
  Telegram-пользователь, тот же принцип, что у `POST /shop/:sellerId/orders`).
- **«Мои заказы» (фронт).** Вместо отдельного экрана деталей — один плоский
  список с полной карточкой заказа инлайн, по образцу уже готового
  `SellerOrders.tsx` (там тот же объём информации не потребовал drill-down).
  Проще и не плодит новый UX-паттерн. Экран `/orders`, без гварда — `AuthProvider`
  уже не рендерит роуты до успешной авторизации (проверено на `CartPage`/
  `ProductDetail`), отдельная проверка не нужна. Иконка вкладки — `ClipboardList`
  (та же, что уже использована для «Заказы» в `AdminToolbar.tsx`, визуальная
  консистентность между поверхностями).
- **FK-индексы.** `index()` — тот же стабильный API `drizzle-orm/pg-core`, что
  уже используется в схеме для `uniqueIndex()` (0.45.2/drizzle-kit 0.31.10,
  синтаксис не менялся годами, доп. ресёрч не требуется). Уточнение к списку
  бэклога: `subscriptions.seller_id` уже под `.unique()` — Postgres сам создаёт
  индекс на unique-констрейнт, отдельный `.index()` там был бы дублем. Реальный
  список новых индексов: `orders.seller_id`, `orders.buyer_telegram_id` (новый,
  под эндпоинт выше), `products.seller_id`, `order_items.order_id`,
  `product_variants.product_id`, `product_images.product_id`,
  `subscription_payments.subscription_id`.
- **Границы Zod-полей.** `items.max(100)` — прямое зеркало `variants.max(10)`.
  `reason` в `updateSellerStatusRequestSchema` (platform.ts:46, БД-колонка
  `blocked_reason`) — `.max(500)`: точного прецедента в CONCEPT нет, выбрано
  как рабочий потолок для короткой админской пометки (короче `buyerComment`
  на 1000, это внутренняя заметка, не текст для покупателя).
- **Идемпотентность теста.** Тот же паттерн, что в `products.route.test.ts`
  (`inArray`/удаление продавцов перед тестом), адаптированный под
  `product-images.route.test.ts`: там `tgCounter` — счётчик, а не
  фиксированный список id, поэтому чистка — `afterAll` с `gte(sellers.telegramId, 700450000)`
  (стартовое значение счётчика), а не `inArray` по константам.
  **Уточнение по факту прогона** (не было видно на этапе анализа): голый
  `gte` без верхней границы задевал и продавцов из **других** файлов с
  telegram_id выше 700450000 (включая новый блок тестов `GET /orders/mine`
  этого же спринта, 700600001+) — у части из них есть заказы, а
  `orders.seller_id` осознанно без `onDelete cascade` (152-ФЗ, см.
  анализ ревью 21.07), так что `DELETE` падал на FK. Исправлено —
  верхняя граница `lte(sellers.telegramId, tgCounter)` (значение счётчика
  на момент `afterAll`, т.е. только реально свои продавцы). Проверено
  тремя прогонами `pnpm test` подряд без ручного `TRUNCATE` между ними —
  зелено каждый раз.

Задачи:

- [x] Бэкенд: эндпоинт «мои заказы» — список заказов покупателя по
  `request.user.telegramId` (не по `sellerId`, как устроены продавцовские
  роуты), плюс, вероятно, детали/статус заказа. Новый метод в
  `orders.service.ts` (`listBuyerOrders`) + роут; ownership — по
  `buyer_telegram_id`, чужие заказы в выборку не попадают. Zod-контракт
  ответа в `packages/shared`.
- [x] Фронт: экран истории заказов покупателя (и деталей заказа), третья
  вкладка «Заказы» в floating toolbar (`FloatingToolbar`/`TabBar`) —
  довешивается к паттерну Спринта 33.
- [x] FK-индексы (пункт `[P1]` бэклога) — `orders.seller_id`,
  `orders.buyer_telegram_id` (новый, под эндпоинт выше),
  `products.seller_id`, `order_items.order_id`,
  `product_variants.product_id`, `product_images.product_id`,
  `subscriptions.seller_id`, `subscription_payments.subscription_id`.
  `.index()` в `db/schema.ts` + миграция `drizzle-kit generate`.
- [x] Границы Zod-полей (пункт `[P2]` бэклога): `items.max(100)` в
  `createOrderRequestSchema`, `blockedReason.max(...)` в platform-схеме.
- [x] Фикс не-идемпотентного `product-images.route.test.ts` (пункт `[P2]`
  бэклога) — очистка посеянных строк (`afterAll` по диапазону id, паттерн
  `ALL_TEST_IDS`/`inArray` из других route-тестов), чтобы повторный
  локальный `pnpm test` не падал на `sellers_telegram_id_unique`.
- [x] typecheck + `pnpm -r test` зелёные; визуальная проверка витрины
  (в пределах ограничений среды, см. заметку Спринта 33).

## Обзор

`pnpm -r typecheck` (все три пакета) и `pnpm -r test` — зелёные: 188
тестов `apps/api` (23 файла, включая новые для `GET /orders/mine`), 76
тестов `apps/web`. Идемпотентность `product-images.route.test.ts`
проверена реально, не только по коду — три прогона `pnpm test` подряд
без ручного `TRUNCATE` между ними, все зелёные (детали находки при
проверке — в «Анализ перед стартом» выше, пункт про идемпотентность).

Индексы: реальный список короче исходного пункта бэклога на один —
`subscriptions.seller_id` уже был проиндексирован своим `.unique()`,
отдельный `.index()` не заведён (см. анализ выше). Миграция
`drizzle/0003_ambitious_vindicator.sql`.

Ограничение среды то же, что в Спринте 33: headless-браузер недоступен
в песочнице агента (нет системных зависимостей Chromium, нет
локального Playwright в проекте) — «визуальная проверка» это разбор по
CSS/паттернам (новая вкладка `OrdersPage` использует тот же `pb-28` и
структуру карточек, что уже визуально проверенный `ProductDetail`/
`SellerOrders`), не реальный клик по интерфейсу. Стоит перепроверить
глазами в реальном браузере при следующей возможности
(`pnpm dev` + `docker compose up -d postgres api`).
