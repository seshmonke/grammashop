### Спринт 23: ПДн-долг + дешёвый технический долг

Цель: закрыть два пункта про ПДн из [ревью 21.07.2026](../reviews/2026-07-21.md)
(сценарий удаления данных продавца, шифрование бэкапов) — оба нужно
закрыть до появления реальных продавцов/покупателей — плюс «дешёвый
долг за один заход» оттуда же. Состав согласован в диалоге 21.07.2026
(`/grill-me`), критический путь v1 по-прежнему заблокирован внешне
(ЮKassa-мерчант, юрпакет 152-ФЗ) — это работа вокруг блокера, не по
нему, сознательно (ревью прямо предупреждает не путать одно с другим).

Решения, зафиксированные при согласовании состава:

- **Продавец: обезличивание, не каскад.** По аналогии с уже принятым
  сценарием покупателя (ПДн живут снапшотом в `orders`, «удаление» —
  точечный `UPDATE`, не удаление строки) — для продавца тоже
  обезличивание записи `sellers` на месте, не `DELETE`. Причины:
  консистентность с прецедентом покупателя; каскад потянул бы удаление
  `orders`, а в них — ПДн *покупателей* (чужие данные удалились бы
  побочным эффектом), плюс историю заказов часто нужно хранить отдельно
  для бухгалтерии. `orders.sellerId` без `ON DELETE` — уже не проблема
  при этом решении: строка `sellers` никогда не удаляется, FK не рвётся.
- **Поля под обезличивание**: `fullName`, `phone` (прямая аналогия с
  `buyerFullName`/`buyerPhone`), и **`paymentDetails`** тоже — реквизиты
  перевода, финансовые данные не менее чувствительны, назначения после
  обезличивания продавца у поля не остаётся. `telegramId`,
  `telegramUsername`, `shopName` — не трогаются (аналогично тому, что
  `buyerTelegramId` не обнуляется у покупателя).
- **Механизм** — не голый SQL от руки, скрипт
  `apps/api/src/db/anonymize-seller.ts` (по образцу `seed.ts`/`backup.ts`),
  принимает `sellerId` явным аргументом, делает `UPDATE`. Меньше риска
  необратимой ошибки, чем ручной `UPDATE ... WHERE id = N`.
- **Шифрование бэкапов — GPG, симметричное**, не SSE и не асимметричное.
  SSE не закрывает названный в ревью риск (утечка ключа сервисного
  аккаунта в `.env` на той же VM — SSE прозрачно расшифровывается для
  любого, у кого есть доступ через S3 API, то есть с теми же ключами).
  Асимметричное потребовало бы держать приватный ключ на VM всё равно —
  `restore-check.ts` расшифровывает автоматически, без участия человека
  (systemd timer), так что офф-VM приватный ключ невозможен, а значит и
  бессмысленен здесь. Симметричный пароль-фраза — в отдельной
  переменной `BACKUP_GPG_PASSPHRASE`, не рядом с S3-ключами: узкая
  утечка S3-доступа отдельно от `.env` целиком не даёт прочитать
  бэкапы. Полную компрометацию VM/`.env` целиком это не закрывает —
  ничего не может при требовании к автоматической расшифровке.
- Существующие бэкапы в S3 не шифруются задним числом — ретеншн (7
  дневных + 4 недельных) сам вытеснит их в течение месяца.
- Флакующий тест `products-import.route.test.ts` — осознанно не в этом
  спринте (слабо связан по теме, не блокирует CI — падает только при
  повторных ручных прогонах без пересоздания тестовой БД). Остаётся в
  [backlog.md](backlog.md).
