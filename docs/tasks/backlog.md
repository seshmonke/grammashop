- **Возврат товара покупателем (закон о защите прав потребителей)** —
  заведено 19.07.2026 по запросу пользователя, требование явно
  сформулировано: минимальная ответственность платформы. Сейчас механика
  возврата не спроектирована вообще — ни в
  [CONCEPT.md#каталог-и-заказы](../CONCEPT.md#каталог-и-заказы), ни в схеме
  БД. Отправная точка та же, что уже принята для оплаты: платформа не
  сторона сделки между продавцом и покупателем — деньги идут напрямую,
  без эскроу, без агентского договора (см.
  [CONCEPT.md#юридическая-форма-отношений-с-продавцом](../CONCEPT.md#юридическая-форма-отношений-с-продавцом)).
  Разбор (рабочий черновик, требует проверки юристом, как и весь юрпакет
  152-ФЗ — не финал):
  1. **Юридическая рамка** — дистанционная торговля (продажа через
     мессенджер подпадает под правила дистанционной продажи, ст. 26.1
     Закона о защите прав потребителей + Постановление Правительства
     №2463): покупатель вправе отказаться от товара **надлежащего**
     качества в течение 7 дней после получения без объяснения причин
     (кроме отдельных исключений — например, товар с индивидуально
     определёнными свойствами, если явно указан этот признак при
     покупке), и вернуть товар **ненадлежащего** качества в течение
     гарантийного/разумного срока. Это обязанность продавца как стороны
     сделки — платформа юридически не обязана организовывать возврат,
     но должна явно закрепить это разделение в оферте (текст уже несёт
     общий принцип «платформа — информационный посредник», см.
     `apps/web/public/offer.html`, п.2 — пункт про возврат нужно
     добавить туда же явно, не полагаться на общую формулировку).
  2. **Что платформа обязана дать технически** (в отличие от
     юридической обязанности — вот это уже наш код): вероятно (а) канал
     связи для запроса возврата — тот же паттерн, что и вся
     коммуникация продавца/покупателя, диплинк в Telegram, не встроенный
     чат/арбитраж (см.
     [CONCEPT.md#коммуникация-продавца-и-покупателя](../CONCEPT.md#коммуникация-продавца-и-покупателя))
     — платформа не обязана быть посредником в споре; (б) решить, нужен
     ли отдельный статус заказа под возврат или это вне статусной модели
     вообще (сейчас «новый→оплачен→выполнен/отменён», «отменён» —
     терминальный без отката, см.
     [CONCEPT.md#каталог-и-заказы](../CONCEPT.md#каталог-и-заказы) — возврат
     это про уже выполненный заказ, другая ветка, не переход из
     существующей статусной модели механически).
  3. **Текст в оферте/публичной странице** — явно прописать право
     покупателя на возврат по закону и то, что весь процесс (кто платит
     за обратную доставку, куда физически возвращать товар, кто и когда
     возвращает деньги) — предмет прямой договорённости продавца и
     покупателя, платформа не участвует и не гарантирует исход. Тот же
     юрпакет, что уже в очереди ниже («Юридический пакет»), должен
     закрыть и этот пункт — не отдельный заказ у юриста.
  **Рекомендация по минимальному скоупу v1/v2**: платформа не
  автоматизирует ни возврат денег, ни логистику назад — только даёт
  покупателю канал связаться с продавцом (уже есть, диплинк в личку с
  экрана заказа) и честно фиксирует в оферте право на возврат по закону.
  Главный компромисс — простота против контроля: минимум кода и
  минимум ответственности платформы, но и минимум автоматизации для
  продавца/покупателя (по аналогии с уже принятым решением по доставке
  на Тарифе 1 — вне ТМА, по прямой договорённости сторон).
- **Прод-VM не достаёт до Telegram API — уведомления продавцу не
  доходят** — обнаружено 20.07.2026 при сквозной проверке чекаута
  Спринта 12: заказ оформляется (`POST /shop/:sellerId/orders` → 201),
  задача исправно встаёт в очередь pg-boss (`order-notification`,
  воркер её подхватывает), но `sendMessage` через grammY падает
  `ETIMEDOUT`. Диагностика с самой VM (`89.169.191.173`, и с хоста, и
  изнутри контейнера `api`): `api.telegram.org`, `t.me`, `telegram.org`
  — TCP-соединение не устанавливается вообще (не DNS, не HTTP-уровень),
  при этом обычный интернет с той же VM работает (Google/GitHub/Яндекс
  — 200). Похоже на блокировку Telegram на уровне сети/провайдера для
  российских серверов (Yandex Cloud, `ru-central1`) — известная история
  с блокировками Telegram в РФ, от которых клиентские приложения уходят
  через встроенные обходы (MTProto-обфускация и т.п.), а прямой
  HTTPS-запрос с сервера — нет. Важное следствие: `/auth` при этом
  работает — проверка `initData` идёт локальным HMAC, без сети до
  Telegram, поэтому это не было замечено раньше; любой **другой** будущий
  функционал, требующий исходящих запросов к Telegram API с сервера
  (не только уведомления о заказах), упрётся в то же самое. Решение
  отложено — сначала перепроверить, не временная ли блокировка (curl
  с VM через несколько часов/дней), если стабильно — рассмотреть
  прокси/туннель для исходящих запросов grammY (библиотека поддерживает
  кастомный fetch) и завести отдельным пунктом с выбором конкретного
  прокси-сервиса.

  **Перепроверено 21.07.2026** (Спринт 21, диагностика перед стартом):
  блокировка стабильна, но точнее, чем казалось изначально, — это не
  сплошной блок всей подсети Telegram, а блокировка конкретного IP.
  `api.telegram.org` резолвится в `149.154.166.110` — TCP до него не
  идёт (100% таймаут на десятке попыток, порт 443, и через обычный
  `curl`, и напрямую в TCP). При этом `149.154.167.220` (тоже валидный
  адрес Telegram Bot API, тот же /16) отвечает мгновенно и корректно
  (`curl --resolve api.telegram.org:443:149.154.167.220` → HTTP 404 от
  Bot API за ~150мс). Другие проверенные адреса того же диапазона
  (`149.154.175.50`, `149.154.175.100`) — снова таймаут. Похоже на
  блокировку по конкретным IP/подсетям, а не по домену или всему ASN
  Telegram — значит рабочий адрес может смениться без предупреждения
  (DNS round-robin на стороне Telegram), полагаться на один захардкоженный
  IP нельзя. Меняет рекомендацию по починке: не обязательно внешний
  прокси-сервис — можно попробовать сначала подключение через
  кастомный `fetch`/agent в grammY с ретраем по нескольким известным IP
  Bot API (или явный резолвер на рабочий адрес), это дешевле и без
  внешней зависимости; если адреса будут «плавать» непредсказуемо —
  тогда прокси. Починка по-прежнему не в этом спринте, только диагностика.
- **Механика уведомлений продавцу через бота — продумать отдельно от
  сетевой проблемы выше** — вопрос поднят пользователем 20.07.2026 при
  проверке того же чекаута: единственный продавец в проде сейчас — это
  и есть владелец платформы (`syzrp`), сообщения технически шли в его
  собственный чат с `@grammashopbot`; для произвольного стороннего
  продавца это не было проверено как общий механизм. Разбор:
  - Ограничение Telegram Bot API — бот не может написать пользователю
    первым, только если тот сам хотя бы раз открыл с ботом диалог/Mini

    `@grammashopbot`, условие выполняется самим фактом регистрации —
    специально ничего заводить не нужно, но это предположение явно
    зафиксировано как требование к будущей форме регистрации (см.
    продуктовую карту, пункт 3 «Регистрация и подписка продавца»), а не
    оставлено молчаливым.
  - Кейс «продавец заблокировал бота после регистрации» —
    `sendMessage` вернёт 403, ловится тем же путём, что и сетевые сбои

  - **Пробел без решения**: провал `sendOrderNotification` сейчас нигде
    не виден платформенному админу — только вручную через SQL по

    `Sentry.captureException` не стоит — ошибка просто утекает в
    pg-boss), ни какого-либо алертинга/дашборда. На масштабе 1-10
    продавцов это терпимо (можно проверить вручную по жалобе продавца
    «не пришло уведомление»), но не решение, а временная терпимость —
    зафиксировать как отдельный технический долг, не путать с самой

  п.3 продуктового бэклога). Разобрано подробно, т.к. модерация всё ещё

     152-ФЗ ниже, не отдельный.

     одобрения, самостоятельно не включается: нужно писать персональному

     отвязывает карту от автосписаний, (б) оценку оборота по рекурренту.
     Оферта должна явно называть сумму и период списания. Заодно
     закрывается вопрос механики Тарифа за 1 ₽ (рекуррент 1 ₽/мес vs
     верификация карты — см. пункт из ревью ниже) — уточняется у того же
     менеджера.

### Из ревью 16.07.2026 (см. [reviews/2026-07-16.md](../reviews/2026-07-16.md))

Концепция и продукт:

- **Проверить руками онбординг ЮKassa как тестовый продавец** (какой URL
  магазина указывать на модерации, требования к статусу самозанятого/ИП)
  — переносится на **v2**, не v1. Это онбординг продавца в его
  *собственном* ЮKassa-аккаунте для приёма оплаты от покупателей (Тарифы
  2-3), а не платёжная механика v1: на Тарифе 1 оплата покупателем —
  мануальный перевод, без платёжного шлюза (см.
  [CONCEPT.md#скоуп-v1-mvp](../CONCEPT.md#скоуп-v1-mvp)). ЮKassa в v1 нужна
  только платформе — для приёма собственной подписки с продавца (см.
  [CONCEPT.md#оплата-подписки-продавцом](../CONCEPT.md#оплата-подписки-продавцом)),
  это отдельная и более простая интеграция без модерации магазина
  продавца. Требования СДЭК к выдаче API-ключей — туда же, в v2
  (концепция п. 2).
- **Уточнить механику Free-тарифа за 1 ₽** по документации ЮKassa — для
  собственного мерчант-аккаунта платформы, не продавца. **Направление
  решено 21.07.2026** (`/grill-me`, см.
  [CONCEPT.md#тарифы](../CONCEPT.md#тарифы)): именно рекуррентное
