### Спринт 32: UX блокировки продавца + флаки тестов

Цель: заблокированный продавец при заходе в свою админку видит понятное
сообщение о блокировке (причина + контакт), а не вилку «О платформе /
Открыть магазин»; админ платформы вводит причину при блокировке, а не
просто жмёт confirm без объяснений. Плюс мелкая техническая уборка —
устранить флаки в тестах `apps/api` на общем `telegram_id` между файлами.

Состав согласован в диалоге 22.07.2026. Найдено там же: `resolveAuthContext`
(`apps/api/src/services/auth.service.ts`) ищет продавца с фильтром
`status = "active"`, поэтому заблокированному продавцу приходит
`sellerId: null` — фронт не отличает его от никогда не регистрировавшегося
пользователя и показывает `Fork.tsx`. Решения по открытым вопросам приняты
в диалоге:

- Контакт для разблокировки — Telegram-аккаунт владельца платформы
  (`@syzrp`), статическая константа во фронте, не через API.
- Причина блокировки — свободный текст в форме блокировки (не
  фиксированный список), хранится как есть.

Задачи:

- [x] Новое поле `sellers.blocked_reason` (text, nullable) — миграция
  drizzle-kit.
- [x] `PATCH /platform/sellers/:id/status` — принимает опциональный
  `reason` при переходе в `blocked`, сохраняет в `blocked_reason`; при
  обратном переходе в `active` поле очищается (`null`).
- [x] Различить в ответе `/auth` «не зарегистрирован» и «заблокирован»:
  новое поле `sellerStatus: SellerStatus | null` (переиспользуем enum
  `active`/`blocked` из `packages/shared`) и `blockedReason: string | null`
  в `authResponseSchema`. `resolveAuthContext` ищет продавца без фильтра
  по статусу для заполнения этих двух полей — `sellerId` при этом
  по-прежнему остаётся `null` для заблокированного продавца, вся текущая
  логика доступа (`requireSellerId`, фронтовые гварды в `guards.tsx`) не
  меняется — минимизирует риск регрессии в контроле доступа.
- [x] Форма блокировки в `PlatformHome.tsx` — заменить `window.confirm`
  на модалку/форму со свободным текстовым полем причины (обязательное
  при блокировке, не нужно при разблокировке).
- [x] Экран «Магазин заблокирован» вместо `Fork` — показывает причину
  (`blockedReason`) и контакт (`@syzrp`, константа); встраивается в
  `Landing.tsx` до текущей развилки `sellerId`/`isAdmin`/`Fork`, по
  признаку `sellerStatus === "blocked"`.
- [x] `RegisterForm.tsx` — добавить обработку ошибки мутации (`onError`),
  сейчас 409 (и любой другой сбой `POST /seller/register`) теряется молча
  (unhandled rejection). Защитный фикс независимо от экрана блокировки —
  тот же провал возможен и по другим причинам, не только у заблокированного
  продавца.
- [x] Флаки в `apps/api` тестах на общем `telegram_id` между файлами
  (найдено в Спринте 31: `orders.route.test.ts` и
  `notifications/order-notification.test.ts` используют одно и то же
  значение) — аудит всех telegram_id-констант в `apps/api/src/**/*.test.ts`
  на уникальность, перераспределить пересекающиеся диапазоны.

Реальная работа (аудит, код) не начинается без отдельной команды старта —
состав зафиксирован в диалоге 22.07.2026.

Команда старта дана 23.07.2026.

**Анализ перед стартом:**

- Все задачи разумны, ни одна не меняет и не расширяет скоуп — состав
  подтверждён без изменений, полный стоп не требуется.
- `resolveAuthContext`/`authResponseSchema` — прочитаны
  `apps/api/src/services/auth.service.ts`, `packages/shared/src/schemas/auth.ts`.
  План подтверждается: убрать фильтр по `status` из select, определять
  `sellerId` явно (`status === "active" ? seller.id : null`), добавить
  `sellerStatus`/`blockedReason` в ответ.
- Форма блокировки (задача про модалку) — в `apps/web` нет ни одного
  Dialog/модального компонента (только `@radix-ui/react-slot` под
  `Button`), админки уже используют инлайн-паттерн для доп. ввода перед
  действием (пример — поле «месяцев доступа» рядом с кнопкой льготы в
  `PlatformHome.tsx`). Заводить `@radix-ui/react-dialog` ради одной формы
  избыточно — делаем инлайн-форму (textarea + подтвердить/отмена) в том же
  стиле, без новой зависимости. Способ реализации внутри
  согласованного скоупа, продолжаем без обсуждения.
- Флаки-аудит — прогнан grep всех 9-значных telegram_id-констант по
  `apps/api/src/**/*.test.ts`. Коллизий больше, чем упомянуто в описании
  спринта: помимо известной пары `orders.route.test.ts` /
  `order-notification.test.ts` (`700500001`, плюс `700500099` в обоих
  файлах), тот же `700500001` использовался и в `billing.route.test.ts`, а
  `700600001` пересекался между `platform.route.test.ts` и
  `billing.service.test.ts`. Аудит и так требовал пройти по всем файлам —
  задача не меняется, просто больше правок, чем казалось на этапе
  формулировки.
- Drizzle-миграция — типовая (`text`, nullable), паттерн уже есть
  (`0001_order_idempotency_key.sql`), команда `db:generate` в
  `apps/api/package.json`.

**Обзор:** все задачи выполнены как описано выше, скоуп не менялся по
ходу работы. `apps/web` typecheck и `pnpm -r test` (23 файла/186 тестов
`apps/api`, 12 файлов/76 тестов `apps/web`) — зелёные. При прогоне тестов
обнаружился протухший мусор в `grammashop_test` (строки `sellers` с
2026-07-22, до перераспределения диапазонов) — БД тестовая, вычищена
`TRUNCATE ... RESTART IDENTITY CASCADE` перед финальным прогоном.

При этом всплыла смежная, но отдельная проблема — `product-images.route.test.ts`
использует собственный счётчик telegram_id без очистки после себя, из-за
чего он не идемпотентен между повторными локальными прогонами (в отличие
от кросс-файловой коллизии, которую чинил этот спринт). Не наш скоуп —
вынесено в [backlog.md](backlog.md#наблюдаемость-и-эксплуатация).
