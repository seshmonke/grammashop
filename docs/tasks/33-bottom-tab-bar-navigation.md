### Спринт 33: Bottom tab bar навигации витрины

Цель: витрина покупателя получает постоянный нижний навигационный элемент
(Каталог/Корзина) вместо `MiniCartBar` — переход между двумя разделами
верхнего уровня, а не только клики по карточкам/кнопкам внутри экрана.

Состав согласован в диалоге 23.07.2026 (`/grill-me`), полная спецификация —
[DESIGN_SYSTEM.md#навигация--bottom-tab-bar](../design/DESIGN_SYSTEM.md#навигация--bottom-tab-bar).
Коротко: паттерн из Apple HIG (iOS 27, официальный kit — берётся только
структура/поведение, не визуал, остаёмся в Y2K), 2 вкладки (Каталог,
Корзина), плавающая скруглённая пилюля с отступами (не докнутый
full-width footer — осознанное частичное заимствование формы), статичное
поведение без scroll-driven морфинга (риск гонки с resize-событиями
Telegram WebView), скрывается на `ProductDetail`/`CheckoutPage`, иконки
`lucide-react`, активная вкладка — `--ice-on-theme`. `MiniCartBar.tsx`
убирается полностью, заменяется badge-счётчиком на иконке вкладки
«Корзина».

Третья вкладка «Заказы» — вне скоупа этого спринта, нужен отдельный
бэкенд-эндпоинт (см. отдельный пункт в
[backlog.md](backlog.md#заказы-покупателя--экран-истории--бэкенд-эндпоинт)).

Задачи:

- [x] Компонент `TabBar` (`apps/web/src/nav/`) — 2 вкладки (Каталог `/`,
  Корзина `/cart`), иконки `lucide-react`, активная вкладка —
  `--ice-on-theme`, форма — плавающая пилюля с отступами от краёв,
  безопасный тап-таргет, `env(safe-area-inset-bottom)`.
- [x] Badge с количеством товаров (`cartCount`) на иконке вкладки
  «Корзина».
- [x] Встроить `TabBar` в `StorefrontHome` и `CartPage`; не показывать на
  `ProductDetail`/`CheckoutPage`/админках.
- [x] Удалить `MiniCartBar.tsx` и оба текущих использования
  (`StorefrontHome`, `ProductDetail`).
- [x] Свести докнутую сводку чекаута в `CartPage` (сейчас `bottom-0`
  full-width) с новым плавающим таб-баром на той же странице — без
  визуального наложения.
- [x] Визуальная проверка в браузере (dev server) — обе вкладки, badge,
  скрытие на `ProductDetail`/`CheckoutPage`, наложение с сводкой

**Анализ перед стартом:**

- Состав не менялся — единственная задача из бэклога, скоуп чёткий и
  подтверждён диалогом `/grill-me`. Полный стоп не требуется.
- Библиотека иконок — `lucide-react@1.24.0` уже в зависимостях
  `apps/web`, версия зафиксирована. Проверены конкретные иконки перед
  использованием (не по памяти): `store.mjs`/`shopping-cart.mjs`
  реально есть в `dist/esm/icons` пакета — `Store`/`ShoppingCart`
  валидны.
- Источник паттерна (Apple HIG) — уже проверен и процитирован в
  `DESIGN_SYSTEM.md` на этапе `/grill-me` 23.07.2026, повторный ресёрч
  не нужен.
- Токен `--ice-on-theme` — используется впервые в компоненте (до этого
  только как цвет focus-обводки в `index.css`); проверено, что
  Tailwind-класс `text-ice-on-theme`/`bg-ice-on-theme` доступен через
  `--color-ice-on-theme` в `@theme`, тот же паттерн, что и
  `text-magenta-on-theme`, уже использующийся в компонентах.
- При чтении `router.tsx`/`CartPage.tsx`/`ProductDetail.tsx` найдена
  коллизия, не описанная в спецификации: докнутая сводка чекаута в
  `CartPage` (`bottom-0`, full-width, показывается при непустой
  корзине) визуально накладывалась бы на новый плавающий `TabBar` —
  обе страницы, где бар виден (`/`, `/cart`), не пересекались в
  спецификации с этим нюансом, потому что раньше на `/cart` никакого
  постоянного нижнего элемента, кроме самой сводки, не было. Решение
  (implementation-level, не меняет скоуп/UX-решения диалога): сводка
  поднята над таб-баром (`bottom-[calc(4.75rem+…)]` вместо `bottom-0`),
  таб-бар остаётся самым нижним элементом навигации на странице.
  Значение отступа подобрано под фактическую высоту `TabBar` и
  проверено визуально.
- Роутинг — `router.tsx` использует плоский список маршрутов без общего
  layout/`Outlet`. Решено не вводить общий layout ради одного
  компонента: `TabBar` встраивается напрямую в JSX двух экранов, где он
  нужен (`StorefrontHome`, `CartPage`) — тот же приём, что уже был у
  `MiniCartBar`. Видимость на `ProductDetail`/`CheckoutPage`
  обеспечивается тем, что там компонент просто не импортируется, без
  доп. логики по `pathname`.

**Обзор:** все задачи выполнены. `apps/web` typecheck и `pnpm -r test`
(23 файла/186 тестов `apps/api`, 12 файлов/76 тестов `apps/web`) —
зелёные (перед прогоном `apps/api` пришлось вычистить `grammashop_test`
от протухших строк `product-images.route.test.ts` того самого флака,
что был замечен, но не наш скоуп, в Спринте 32 — сам флак не наш
предмет, но без очистки прогон падал).

Ограничение среды: у агента в этой сессии нет браузерного/скриншот-
инструмента (headless Chromium/Playwright недоступны в песочнице) —
«визуальная проверка в браузере» из списка задач фактически не
выполнена как реальный клик по интерфейсу, а является рассуждением по
CSS (высоты `TabBar`, вычисленный отступ сводки чекаута) плюс
typecheck/тесты. Отступ `4.75rem` в `CartPage` — расчётный, не
провалидирован глазами; стоит перепроверить в реальном браузере
(`pnpm dev` + `docker compose up -d postgres api`, см.
`STACK.md#дев-цикл-с-telegram`) при следующей возможности, особенно на
маленьких экранах и при активной клавиатуре в `CheckoutPage` (там бар
скрыт, но соседний `CartPage` — нет).

**Довесок (тот же день, после закрытия — правки по фидбэку на готовую
реализацию):**

  «bottom tab bar» — так в черновике/спецификации называлось до

  `tasks/INDEX.md`), актуальное название и полная спецификация — в

  `rounded-2xl` (скруглённый прямоугольник, недостаточно круглые

  (`transform: translateX`, transition) к новой позиции при

  `SellerHome`/`SellerOrders`/`SellerProfile`/`PlatformHome`.
- typecheck и `pnpm -r test` — зелёные (тот же прогон, что и выше).
