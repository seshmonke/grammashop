обсуждается отдельно по факту находок этой волны.

## Аудит: находки

1. **«Успех» существует в коде, но не в палитре**: `SellerProfile.tsx:212`
   красит подтверждение сохранения формы в голый `text-emerald-500`.
   Первая гипотеза (цвет — случайная отсебятина) не подтвердилась при
   проверке `grep` по всему `apps/web/src` (не по памяти): тот же голый
   `emerald-500` уже используется в canon-реализации формы
   (`CheckoutPage.tsx` — зелёная обводка валидного тронутого поля,
   `fieldBorderClass`). То есть «успех» как визуальный сигнал в проекте
   уже есть де-факто, просто нигде не токенизирован — отсюда разъезд:
   два независимых места используют один hex-класс мимо системы токенов
   (`--tg-*`), не зная друг о друге. Решение и новый токен —
   см. `DESIGN_SYSTEM.md#палитра`.
2. **Три разных паттерна возврата на предыдущий экран админки**:
   `Button asChild` в шапке (`SellerOrders` → «Товары», `PlatformHome` →
   «Мой магазин»); голая текстовая ссылка со стрелкой над `h1`
   (`SellerProfile` → «← Товары»); полное отсутствие навигации в шапке
   — только кнопка «Отмена» внизу формы (`ProductForm`), то есть чтобы
   покинуть экран редактирования товара, нужно долистать форму до
   конца.
3. **Состояния загрузки/пусто/ошибки не используют `ScreenState`**
   (`apps/web/src/shop/ScreenState.tsx`, введён Спринтом 28 как единый
   паттерн): `SellerHome`, `SellerOrders`, `SellerProfile`,
   `PlatformHome` — у каждого свой инлайн `<p className="py-16
   text-center text-tg-hint">…</p>`, то есть ровно тот же разнобой,
   который Спринт 28 уже устранял на витрине, просто не унаследован
   админками. Все четыре случая — вариант `inline` (шапка уже известна
   и отрисована). Отдельно у `ProductForm` в режиме редактирования нет
   вообще никакого индикатора, пока `existing` разрешается из ещё не
   загруженного списка товаров — форма просто на мгновение рендерится
   пустой, а потом наполняется.
4. **Пустое состояние `SellerHome` без пути дальше**: «Пока нет ни одной
   карточки товара.» — без ссылки на добавление товара, хотя действие
   явно существует рядом (кнопка «Добавить» в шапке) — нарушает правило
   Спринта 28 «путь дальше, если он объективно существует». Пустые
   состояния `SellerOrders`/`PlatformHome` корректно без действия — у
   продавца/платформы объективно нечего делать со списком без записей.
5. **Деструктивные действия — два разных визуальных языка без
   зафиксированного правила**: голый `text-tg-destructive` текстом
   (`ProductForm`: «Убрать вариант», «Удалить фото») против нейтрального
   `Button variant="ghost"` без всякого намёка на цвет (`SellerHome`:
   «Удалить» карточку товара; `SellerOrders`: «Отменить заказ» —
   единственный необратимый переход по комментарию в самом коде;
   `PlatformHome`: «Заблокировать») — для действий сопоставимой тяжести.
   Отдельно: «Удалить фото» в `ProductForm` — единственное необратимое
   действие среди всех шести экранов без `confirm()` (удаление товара,
   отмена заказа, блокировка продавца — все с подтверждением).
6. **Три формы всё ещё на старом паттерне** (canon зафиксирован Спринтом
   22, уже отмечено как «Не реализовано» в `DESIGN_SYSTEM.md`, но не
   выполнено попутно — этот спринт явно поглощает бэклог-пункт):
   - `RegisterForm` — submit дизейблится по невалидности (`!canSubmit`),
     прямое нарушение канона; кнопка — свой `<button>` вместо общего
     `Button`, тогда как `ProductForm`/`SellerProfile` используют
     `Button` — третий разнобой компонентов для одной и той же роли
     (magenta CTA); нет блюр-валидации и построчных ошибок вообще.
   - `ProductForm` — submit уже корректно не дизейблится по
     невалидности (только `isPending`, менять не нужно); но ошибка —
     одна общая строка под формой, не под конкретным полем; блюр-
     валидации нет; необязательные поля (описание, базовая цена,
     остаток) помечены только через placeholder, не через суффикс
     `(необязательно)` в лейбле, как того требует канон.
   - `SellerProfile` — submit уже корректно не дизейблится по
     невалидности (менять не нужно); необязательные поля (описание
     магазина, реквизиты) вообще без пометки.

**Рассмотрено и НЕ найдено расхождением**: минимальный Y2K-ДНК (Спринт
19) — заголовок `y2k-heading font-display` и magenta на главной
созидающей кнопке экрана («Добавить»/«Сохранить»/«Оплатить подписку»)
соблюдены на всех шести экранах без исключений; экраны без такой
кнопки (`SellerOrders`, `PlatformHome` — списочные/модерационные, не
создающие) корректно её не имитируют.

## Итог

Правила закреплены в `docs/design/DESIGN_SYSTEM.md`: новый раздел
«Навигация назад», расширение раздела «Состояния» (`ScreenState` —
общий компонент проекта, не только витрины), два новых уточнения в
«Палитре» (нейтральная окраска деструктивных действий, токен
`--tg-success`), формы `RegisterForm`/`SellerProfile`/`ProductForm`
переведены на канон раздела «Формы» (Спринт 22).

Код: `index.css` (токен `--tg-success`), `lib/field-styles.ts` (новый,
общая функция рамки поля), `seller/validate-register-form.ts` (новый,
Zod-валидация той же схемой, что и бэк, по образцу
`checkout/validate-checkout-form.ts`), `shop/ScreenState.tsx`
(переиспользован без изменений), `routes/seller/SellerHome.tsx`,
`ProductForm.tsx`, `RegisterForm.tsx`, `SellerOrders.tsx`,
`SellerProfile.tsx`, `routes/platform/PlatformHome.tsx` — все находки
закрыты.

Вне скоупа (зафиксировано отдельным пунктом в
[backlog.md](backlog.md)): перевод `CheckoutPage.tsx` (не админка) с
голого `border-emerald-500` на новый токен `--tg-success` — то же
значение, чисто косметическая замена класса.

Проверено вживую в браузере (Playwright headless, dev-режим авторизации,
сидовый продавец + отдельный незарегистрированный dev-пользователь для
`RegisterForm`): все шесть экранов, блюр-валидация с подсветкой рамки
и инлайн-ошибками на `RegisterForm`/`ProductForm`, submit не
дизейблится по невалидности, `ScreenState` на состояниях
загрузки/пусто/ошибки, нейтральный (не destructive) стиль «Убрать
вариант»/«Удалить фото», цвет «Сохранено» на `--tg-success`, единый
паттерн кнопки «назад» в шапке. Light/dark тема Telegram отдельно не
проверялась — вне реального Telegram-клиента `--tg-theme-*` не
инжектятся, а изменения этого спринта токонов светлой/тёмной темы не
касаются (только фиксированные `--tg-destructive`-аналоги и структура).
