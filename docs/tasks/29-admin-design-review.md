### Спринт 29: Ревью дизайна админок (волна 2 — продавец + платформа)

Цель: продавцовская (`SellerHome`, `ProductForm`, `RegisterForm`,
`SellerOrders`, `SellerProfile`) и платформенная (`PlatformHome`) админки
единообразны и следуют явным правилам `docs/design/DESIGN_SYSTEM.md` —
в первую очередь минимальному Y2K-ДНК (Спринт 19: Unbounded на
заголовках + magenta на главной созидающей кнопке экрана, без стекла/
scan-lines/полного рескина) и канону форм (Спринт 22: один столбец,
пометка необязательных, blur-валидация с инлайн-ошибками, submit не
дизейблится по невалидности).

Вторая волна зонтичной задачи «ревью дизайна всего проекта» (Спринт 28 —
волна 1, витрина покупателя). Публичный лендинг — волна 3+, состав
обсуждается отдельно по факту находок этой волны.

**В скоуп поглощён бэклог-пункт** [P2] «Перевести формы на канон из
DESIGN_SYSTEM.md#формы» — `RegisterForm.tsx`, `SellerProfile.tsx`,
`ProductForm.tsx` сейчас на старом паттерне (дизейбл submit до
валидности, общая ошибка без разбивки по полям, без пометки
необязательных полей). Тот же файл и та же тема, что и остальной аудит
волны — не открывать дважды.

**Явно исключено из скоупа** (нерешённые продуктовые вопросы из
`docs/tasks/backlog.md`, раздел «Продукт — требует обсуждения до кода»,
даже если тема всплывёт в `ProductForm`): категории товаров в каталоге,
кадрирование фото товара, пропорции изображения в карточке/каталоге.

**Граница держим фиксированной, не пересматриваем**: «минимальный Y2K-ДНК
админок» (Спринт 19) — аудит проверяет соблюдение этой границы (не
предлагает добавить админкам стекло/scan-lines/полный акцент).

Порядок внутри спринта — сначала весь аудит и вся правка
`DESIGN_SYSTEM.md`, потом весь рефакторинг кода (как в Спринте 28, не
вперемешку).

Задачи:

- [ ] Аудит всех шести экранов админок — чек-лист конкретных находок:
  соблюдение минимального Y2K-ДНК (заголовки/главная кнопка), отклонения
  трёх форм от канона (`RegisterForm`/`SellerProfile`/`ProductForm`),
  непоследовательные пропорции/отступы/паттерны состояний между
  экранами (по аналогии со Спринтом 28).
- [ ] По каждой находке — правило в `DESIGN_SYSTEM.md` (явная норма +
  обоснование, где уместно — ссылка на источник, не на память).
- [ ] Рефакторинг всех шести экранов по зафиксированным правилам,
  включая перевод трёх форм на канон.

Реальная работа (аудит, правки, код) не начинается без отдельной команды
старта — состав зафиксирован в диалоге 22.07.2026.
