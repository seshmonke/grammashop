# Задачи

Скрам-доска проекта: очередь → спринт → в работе → выполнено. Когда
задача решена, итоговое решение уходит в чистовик (`CONCEPT.md`/
`STACK.md`), а здесь остаётся короткая запись со ссылкой — этот файл про
статус, не про содержание решений.

## Воркфлоу спринтов

Работа идёт спринтами по **1 неделе** (пн–вс). Соло-проект, поэтому без
лишних ролей и церемоний — просто ритм и явные границы «что делаем
сейчас».

1. **Бэклог** — раздел «Очередь» ниже. Весь пул задач без привязки к
   времени, не отсортирован по приоритету жёстко.
2. **Планирование** — в начале недели открываем новый подраздел в
   «Текущий спринт»: `### Спринт N (дд.мм–дд.мм): цель`. Цель — одна
   фраза, что должно быть верно к концу недели. Под ней — список задач,
   перенесённых из «Очередь» (столько, сколько реально сделать за
   неделю, не больше).
3. **В работе** — задача, которую делаем прямо сейчас. Не больше
   1-2 одновременно, чтобы не размазываться.
4. **Дейлик** — короткая сверка в начале каждого рабочего дня (не
   встреча, соло-проект — просто пройтись по доске перед стартом):
   - что сделано со вчера → отметить/перенести в «Выполнено»;
   - что берём в работу сегодня → обновить «В работе»;
   - что новое всплыло (в диалоге, в коде, где угодно) → сразу
     завести задачей в «Очередь», а если это горит и укладывается в
     цель спринта — сразу в список текущего спринта.
   Смысл дейлика — не дать новым задачам потеряться и не копить их «в
   уме»: всё, что всплыло, в тот же день попадает в файл.
5. **Обзор и ретро** — в конце спринта:
   - сделанное → в «Выполнено» со ссылкой на раздел чистовика;
   - несделанное → обратно в «Очередь» (или явно в следующий спринт,
     если это по-прежнему приоритет);
   - при необходимости — короткая заметка в самом подразделе спринта:
     что мешало, что стоит изменить в следующем.
6. Подраздел закрытого спринта остаётся в файле как история (не
   удаляется), открывается следующий.

Если ритм окажется неудобным (например, задачи по концепции решаются не
пачками, а по одной в диалоге) — длину спринта или сам формат можно
пересмотреть, это рабочий процесс, а не зафиксированное решение.

## Дорожная карта проекта

Высший масштаб планирования: инженерный трек от пустого репозитория до
работающего деплоя на сервере. Это не про фичи продукта (подписки,
каталог, чекаут и т.д. — они появятся как задачи внутри веток «Бэкенд»/
«UX/UI» позже) и не про юридическо-организационные вещи (оферта,
юрлицо, пилотный запуск) — те заводятся отдельными задачами, когда до
них дойдёт очередь. Здесь — только базовые инструменты и порядок, в
котором они поднимаются, чтобы можно было работать на следующем уровне
абстракции. Веха 1 — фундамент; 2-4 — следующий слой, может идти
параллельно; 5-6 — финал. Редактируется по ходу работы.

- [ ] **1. Инициализация проекта** — монорепо (pnpm workspaces), базовая
  структура `apps/web`/`apps/api`/`packages/shared`, `package.json` и
  TypeScript-конфиги в каждом пакете, `.env.example` и схема
  обязательных переменных окружения (БД, JWT-секрет, ключи S3 и т.д.),
  минимальный `.gitignore`.
- [ ] **2. Докер и локальное окружение** — `docker-compose.yml`
  (postgres + тестовая БД/схема + api + web), multi-stage `Dockerfile`
  на каждый сервис, `docker compose up` поднимает всё локально и
  отдаёт "hello world" с api и web.
- [ ] **3. База данных** — подключение Drizzle к Postgres, первая
  миграция, базовая схема (заглушка), проверка миграций в докере.
- [ ] **4. Бэкенд** — скелет Fastify, пайплайн `routes → services → db`
  (см. `STACK.md#пайплайн-запроса`), Zod-схемы в `packages/shared`,
  первый health-check эндпоинт — по TDD (см. `STACK.md#тестирование`):
  сначала падающий тест через `fastify.inject()`, потом код.
- [ ] **5. UX/UI** — скелет React + Vite, `shadcn/ui` подключён,
  роутинг трёх групп маршрутов (см. `STACK.md#роутинг`), axios
  настроен на обращение к `apps/api`, первый компонент — тоже по TDD
  (`@testing-library/react` + Vitest).
- [ ] **6. CI/CD** — GitHub Actions: прогон тестов (Vitest, красные —
  пайплайн падает) → сборка Docker-образов → пуш в registry → SSH на
  VM → `docker compose pull && up -d`.
- [ ] **7. Деплой на сервер** — VM в Yandex Cloud, `docker-compose` в
  проде с Caddy как reverse proxy (автоматический TLS через
  Let's Encrypt — обязателен, ТМА открывается только по HTTPS),
  S3-хранилище и Sentry подключены и проверены на бою.
