# Черновик

Свободная форма, рабочая площадка для мыслей о проекте. Всё, что здесь есть,
не окончательно — можно противоречить самому себе, накидывать варианты,
не заморачиваться структурой. Как только мысль устоялась — переносим её в
чистовик (`CONCEPT.md` или `STACK.md`).

---

Изначально была идея сделать магазин чёток в Telegram. Но потом подумали —
а почему бы не сделать так, чтобы любой продавец мог запустить свой магазин
в Telegram, не разрабатывая с нуля свою Mini App. То есть магазин чёток —
это просто первый продавец на платформе, а не сам продукт. Делаем платформу
сразу, "Shop-as-a-Service".

Как это должно работать (в общих чертах): продавец подписывается на
платформе, получает ссылку типа `t.me/platform_bot/app?startapp=seller_id`,
кидает её к себе в канал/бот. Покупатель кликает — открывается общий ТМА
платформы, который рисует витрину именно этого продавца по его id. Внутри —
полноценная корзина, чекаут, оплата. Всё как один бот с мультитенантностью
на уровне данных, а не N ботов на каждого продавца — это важно, чтобы не
плодить сущности и не усложнять деплой.

Монетизация — только подписка с продавца. Никакой комиссии с продаж —
подумали, что это проще и понятнее продавцу, плюс платформа не участвует в
деньгах покупателя, меньше головной боли с эскроу и юридической
ответственностью за чужие деньги.

Платежи — деньги покупателя идут напрямую продавцу. Продавец сам вводит
свои ключи от, например, ЮKassa. Не любая платёжка — ограниченный список,
чтобы не превращать интеграцию в бесконечный саппорт. С оплатой самой
подписки продавцом пока не до конца понятно как — надо думать отдельно
(может это отдельный флоу оплаты через какой-то один способ, может что-то
ещё).

Доставка — тоже даём выбор: либо продавец сам договаривается с покупателем
(руками, в переписке), либо подключает СДЭК через свои же ключи. Важно —
платформа не сторона договора с СДЭК, это между продавцом и СДЭК. Тут ещё
не проработано с юридической и технической стороны, надо будет
разбираться отдельно, когда дойдём до этой части.

Кастомизация витрины — хочется, чтобы магазин не выглядел как одна и та же
шаблонная страница у всех продавцов. Цвета, баннеры, расположение блоков —
чтобы ощущалось как "свой" магазин.

Модерация — решили не делать предварительную проверку карточек товара
(долго, дорого, не масштабируется на старте). Вместо этого — просто лимиты
на количество товаров на продавца, чтобы если кто-то захочет мусорить —
упёрся в потолок быстро. Не идеальное решение, но норм для старта.

Стек в общих чертах обсуждали так: фронт — React + TS + Vite, ну тут
особо вариантов не рассматривали, самое живое и стандартное. Бэк — думали
между Express и Fastify, выбрали Fastify — валидация из коробки через Zod,
меньше писать руками. База — Postgres, ORM — думали Prisma vs Drizzle,
в итоге Drizzle, потому что не тащит свой query-engine бинарник и миграции
это просто читаемый sql, а не сгенерированная магия непонятная. Минус —
Drizzle меньше туториалов, но решили что это ок, разберёмся по докам.

Валидацию (Zod-схемы) решили держать в общем пакете, шарить между фронтом
и бэком, чтобы не писать валидацию дважды в двух местах.

Монорепа — pnpm workspaces, стандартная раскладка apps/web, apps/api,
packages/shared.

Докер — compose для локальной разработки, отдельные multi-stage
Dockerfile-ы под прод на каждый сервис.

Тесты — Vitest, один раннер и для фронта, и для бэка, чтобы не тащить два
разных тестраннера.

Общий принцип по архитектуре — не переусложнять заранее. Простая слоистая
схема routes → services → db, без DDD и прочего наворота, пока не появится
реальная причина это городить.
