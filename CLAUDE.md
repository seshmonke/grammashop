# grammashop

Telegram Shop-as-a-Service — платформа для запуска магазинов в Telegram по
подписке. Концепция проработана (`docs/CONCEPT.md`), инженерный скелет
(монорепо, докер, CI/CD, прод-VM) поднят — идёт работа над продуктовыми
фичами v1.

## Документация (`docs/`)

- `docs/CONCEPT.md` — чистовик, концепция продукта. Только принятые
  решения, формулировки окончательные, без "возможно"/альтернатив.
- `docs/STACK.md` — чистовик, технологический стек.
- `docs/DRAFT.md` — черновик. Сырые мысли, наброски, варианты — до того
  как решение принято. Можно противоречить самому себе, порядок не важен.
- `docs/tasks/` — скрам-доска для отслеживания задач по проработке
  концепции и проекта в целом. `docs/tasks/INDEX.md` — правила
  воркфлоу и список спринтов; один файл `docs/tasks/NN-краткое-описание.md`
  на спринт (история закрытых спринтов не копится в одном файле);
  `docs/tasks/backlog.md` — бэклог («Очередь»); `docs/tasks/milestones.md`
  — «Выполнено» (концептуальные решения, закрытые попутно со спринтами).

Какой факт где живёт (черновик vs чистовик, граница CONCEPT/STACK, судьба
тематических черновиков вроде `premium-features.md`) и правило переноса
описаны в самом `docs/CONCEPT.md` (раздел «Какой факт где живёт»).

## Воркфлоу работы над концепцией

Когда обсуждаем очередной открытый вопрос по продукту:

1. Не просто перечисляй варианты — давай рекомендацию с обоснованием и
   явным главным trade-off'ом, решение всё равно за пользователем.
2. Как только решение принято в диалоге — сразу фиксируй его в
   соответствующем чистовике (`CONCEPT.md` или `STACK.md`), убирай из
   «Открытые вопросы», если пункт там был.
3. Отмечай задачу выполненной в файле текущего спринта (`docs/tasks/`) —
   и/или в `docs/tasks/milestones.md`, если решение концептуального
   уровня, со ссылкой на раздел чистовика — и/или заводи новые пункты в
   `docs/tasks/backlog.md`, если в процессе обсуждения всплыли следующие
   шаги (например, «согласовать текст оферты с юристом»).
4. Если решение предварительное/рабочее (например, цена, которая ещё
   будет пересчитана) — фиксируй его в чистовике, но явно помечай как
   рабочий ориентир, а не финал.

## Git

После закрытия каждого спринта (все задачи в файле спринта
`docs/tasks/NN-*.md` отмечены выполненными; для спринтов-вех — плюс
закрыта веха дорожной карты, промежуточные корректирующие спринты вроде
`23.5` веху не закрывают) — закоммитить и запушить в `origin/main`.

После этого пуша задачи для следующего спринта не формулировать
самостоятельно — сначала обсудить и согласовать состав с пользователем
в диалоге, и только потом фиксировать спринт новым файлом
`docs/tasks/NN-краткое-описание.md` + запись в `docs/tasks/INDEX.md`.

Обсуждение задач следующего спринта начинать сразу после закрытия
предыдущего, не откладывая — для этого можно использовать команду
`/grill-me`. Но фиксация состава в диалоге — это ещё не разрешение
приступать: реальную работу (код, коммиты) над задачами следующего
спринта не начинать без команды старта от пользователя («начинаем
спринт» и т.п.). Между фиксацией состава и командой старта ничего не
делать. Если команда старта прозвучала, а состав ещё не согласован —
задачи не придумывать самостоятельно, начать с обсуждения состава.

Команда старта — полное разрешение на работу над спринтом, отдельно
просить анализ не нужно. Первым шагом после неё, до любого кода, —
анализ состава: по каждой задаче проверить, что она разумна и не
навредит проекту дальше; по задачам, где есть выбор способа реализации
(технология, UX-паттерн, архитектура), — поискать в интернете, как эту
проблему решают другие, и сверить наш подход с лучшими решениями; по
библиотекам и инструментам, которые будут использоваться в реализации
задач спринта, — просмотреть актуальную документацию (API, версии,
ограничения), а не полагаться на память. Результаты анализа принести в
диалог. Дальше:

- Анализ чистый или меняет только способ реализации внутри
  согласованного скоупа (библиотека, паттерн) — сразу продолжать
  работу, взяв лучший вариант из ресёрча и зафиксировав его в заметке.
- Анализ опрокинул или изменил хотя бы одну задачу (убрать, разбить,
  существенно расширить, либо всплыло решение уровня концепции —
  деньги, ПДн, внешние сервисы) — полный стоп всего спринта, даже если
  остальные задачи чистые: сначала правка состава с пользователем,
  потом новая команда старта.

Выводы фиксируются как обычно (чистовики, текст задач), плюс короткая
заметка «Анализ перед стартом» в файле спринта (`docs/tasks/NN-*.md`) —
пара строк на задачу, где был ресёрч. Задача, добавленная в текущий
спринт на ходу (через сверку), проходит ту же проверку в лёгкой форме
до написания кода по ней; заметка дополняется.

## Дисциплина работы с ПДн (152-ФЗ)

Платформа — оператор персональных данных (ФИО, телефоны, адреса
покупателей и продавцов), полные требования — в
`docs/CONCEPT.md`, раздел «Персональные данные (152-ФЗ)». Правила для
любого кода и конфигов в этом репозитории:

- Не логировать ПДн: тела запросов чекаута/регистрации, ФИО, телефоны,
  адреса не попадают ни в pino, ни в вывод ошибок.
- В Sentry — только после scrubbing: события не содержат тел запросов и
  полей с ПДн (серверы Sentry за пределами РФ — это трансграничная
  передача).
- ПДн не отправляются во внешние сервисы за пределами РФ. Любая новая
  интеграция, получающая ПДн, — сначала проверка юрисдикции и явное
  решение в диалоге, потом код.
- Новые поля с ПДн в схеме БД или формах не добавляются «про запас» —
  только с назначением, зафиксированным в `CONCEPT.md`.
- Дампы и выгрузки прод-БД не сохранять в scratchpad, `/tmp` или
  репозиторий — бэкапам место только в предусмотренном S3-бакете
  (`STACK.md#бэкапы`).

## Стиль общения

Без формальностей и подлизывания. Не открывай ответ согласием или похвалой
идеи — сразу по делу. Если идея слабая, шаткая или есть более сильный
вариант — прямо говори об этом и объясняй, почему, а не подсвечивай это
вскользь после реверансов. Не нужно смягчать оценку ради вежливости и не
нужно занижать её из осторожности — оценка должна быть точной. Это
касается любого обсуждения, не только вызовов `/balanced`.

Когда нужно что-то уточнить у пользователя — не задавай открытый вопрос
текстом, а предлагай варианты ответа (инструмент для уточняющих вопросов),
всегда с возможностью указать свой вариант вместо предложенных.
